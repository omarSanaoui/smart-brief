Backend Deep Dive

Node.js · Express · JWT · bcrypt · Passport · Zod · microservices. Everything in the Smart Brief backend — how it works, why it's built this way, and what every line means.

Node.js & the Runtime

Node.js is a JavaScript runtime built on Chrome's V8 engine. It lets you run JavaScript outside the browser — on a server. Before Node.js, JavaScript only ran in browsers. Now the same language powers both the React frontend and the Express backend.

The key characteristic of Node.js is that it is single-threaded and non-blocking. It does not create a new OS thread for every incoming request (like PHP or Java). Instead, it uses an event loop — when an operation like a database query or HTTP call is waiting, Node hands control back and handles other requests. When the result arrives, it resumes.

Blocking (other languages)
Thread 1: waiting for DB query...
          (blocked, can't do anything)
Thread 2: serving another request
Thread 3: waiting for file read...
          (blocked)
→ need many threads to handle load
Node.js (non-blocking)
Single thread:
→ start DB query (async)
→ handle another request
→ handle another request
→ DB result arrives → resume first
→ one thread handles everything

This is why all database calls in the project use async/await — they are non-blocking I/O operations.

Express.js

Express is a minimal web framework for Node.js. Node's built-in http module can create a server, but it is very low-level — you would have to manually parse URLs, headers, and JSON. Express wraps all of that and gives you a clean API for:

  • Defining routes (router.get("/path", handler))
  • Running middleware in sequence
  • Parsing request bodies (express.json())
  • Sending responses (res.json(), res.status(404).send())
// The minimal shape of every Express app in this project
import express from "express";
const app = express();

app.use(express.json());            // parse JSON request bodies
app.use("/auth", authRoutes);       // mount a router

app.listen(3000, () => console.log("Running"));

Request and Response objects

Every route handler receives req (the incoming request) and res (your tool to send back a response):

// What lives on req:
req.body        // parsed JSON body (needs express.json() middleware)
req.params      // URL params → /users/:id → req.params.id
req.query       // query string → /users?role=ADMIN → req.query.role
req.headers     // all HTTP headers including Authorization
req.method      // "GET", "POST", "PATCH", "DELETE"
req.path        // "/auth/login"
req.user        // set by authenticate middleware (custom)

// What you call on res:
res.json({ message: "ok" })       // send JSON, sets Content-Type header
res.status(400).json({ ... })     // set status code then send
res.status(204).send()           // 204 No Content (after DELETE)
res.redirect("https://...")       // 302 redirect
res.setHeader("Content-Type", "application/pdf")
res.send(pdfBuffer)              // send binary data

Request Journey — End to End

Let's trace what happens when the admin clicks "Accept" on a brief:

1. Frontend dispatches updateBriefThunk
   → Axios sends: PATCH https://gateway.railway.app/briefs/admin/abc123/status
     Headers: Authorization: Bearer eyJhbGci...
     Body:    { status: "ACCEPTED" }

2. Gateway receives the request
   → CORS check: is the origin allowed? ✓
   → Is it a public route? No
   → authenticate() runs:
       - extracts "eyJhbGci..." from Authorization header
       - jwt.verify(token, JWT_SECRET) → { id: "user-id", role: "ADMIN" }
       - adds req.headers["x-user-id"] = "user-id"
       - adds req.headers["x-user-role"] = "ADMIN"
   → createProxyMiddleware forwards to Brief Service:
       PATCH http://brief-service:4000/briefs/admin/abc123/status
       (same headers, including x-user-id and x-user-role)

3. Brief Service receives the request
   → requireAuth middleware reads x-user-id + x-user-role
   → sets req.user = { userId: "user-id", role: "ADMIN" }
   → route matches: PATCH /admin/:briefId/status
   → updateBriefStatusHandler runs:
       - checks auth.role === "ADMIN" ✓
       - validates body with Zod
       - calls briefService.updateBriefStatus(...)
       - sends notification email
       - res.json(updatedBrief)

4. Response travels back through the proxy to the frontend
5. Redux reducer updates state.briefs.currentBrief
6. Component re-renders with new status badge

Gateway — Overview

The gateway is the simplest of the three services. It has no database. Its only job is: receive requests, check the JWT, forward to the right service.

gateway/src/index.ts

import express, { Request, Response, NextFunction } from "express"
import { createProxyMiddleware } from "http-proxy-middleware"
import jwt from "jsonwebtoken"
import cors from "cors"

dotenv.config()   // load .env file into process.env

const app = express()
const JWT_SECRET = process.env.JWT_SECRET as string

CORS — Cross-Origin Resource Sharing

Browsers block requests from one origin to another by default. A request from https://smartbrief.vercel.app to https://gateway.railway.app is cross-origin — the browser will refuse it unless the server explicitly says it's allowed via CORS headers.

The gateway uses a dynamic origin check instead of a static list, so any Vercel preview deployment (*.vercel.app) also works:

gateway/src/index.ts
const allowedOrigins = [
    process.env.FRONTEND_URL || "http://localhost:5173",
    "http://localhost:5173",
]

app.use(cors({
    origin: (origin, callback) => {
        // origin is undefined for server-to-server calls (no browser involved)
        if (!origin) return callback(null, true)

        // exact match OR any vercel preview
        if (allowedOrigins.includes(origin) || origin.endsWith(".vercel.app")) {
            callback(null, true)   // ✓ allow
        } else {
            callback(new Error("Not allowed by CORS"))  // ✗ block
        }
    },
    credentials: true   // allow cookies/auth headers cross-origin
}))
Why is CORS on the gateway, not the services? The gateway is the only service the browser talks to. The auth and brief services are internal — the browser never reaches them directly. So CORS only needs to be configured once, on the gateway.

JWT Verification at the Gateway

For every non-public request, the gateway reads and verifies the JWT. If valid, it extracts the user's id and role and injects them as headers before forwarding:

gateway/src/index.ts
function authenticate(req: Request, res: Response, next: NextFunction) {
    const isPublic = publicRoutes.some(
        r => req.path.startsWith(r.path) && req.method === r.method
    )
    if (isPublic) return next()

    // Extract token from: "Authorization: Bearer eyJhbGci..."
    const token = req.headers.authorization?.split(" ")[1]
    if (!token) {
        res.status(401).json({ message: "No token provided" })
        return
    }

    try {
        // Verify signature and expiry — throws if invalid
        const decoded = jwt.verify(token, JWT_SECRET) as { id: string, role: string }

        // Attach identity to the forwarded request as trusted headers
        req.headers["x-user-id"]   = decoded.id
        req.headers["x-user-role"] = decoded.role
        next()
    } catch {
        res.status(401).json({ message: "Invalid or expired token" })
    }
}

app.use(authenticate)  // runs before every route below this line

Reverse Proxy with http-proxy-middleware

After authentication passes, the gateway forwards the request to the appropriate service. It does not re-implement any business logic — it is purely a traffic director.

gateway/src/index.ts
// All /auth/* requests → Auth Service
app.use("/auth", createProxyMiddleware({
    target: process.env.AUTH_SERVICE_URL || "http://localhost:3000",
    changeOrigin: true,
    // pathRewrite explains what happens to the path:
    // Gateway receives:  /auth/login
    // Express strips:    /auth  (because app.use("/auth", ...))
    // path is now:       /login
    // pathRewrite adds:  /auth back → /auth/login
    // Auth service gets: /auth/login  ✓
    pathRewrite: (path) => `/auth${path}`,
}))

// All /briefs/* requests → Brief Service (same pattern)
app.use("/briefs", createProxyMiddleware({
    target: process.env.BRIEF_SERVICE_URL || "http://localhost:4000",
    changeOrigin: true,
    pathRewrite: (path) => `/briefs${path}`,
}))

// Internal route for service-to-service (brief-service fetching user info)
app.use("/api/users/internal", createProxyMiddleware({
    target: process.env.AUTH_SERVICE_URL || "http://localhost:3000",
    changeOrigin: true,
}))

x-user-id / x-user-role Headers

This is the security contract between the gateway and the backend services. The gateway is the only thing that can set these headers after verifying the JWT. The backend services trust them completely — they never re-verify the token themselves.

Frontend
Authorization: Bearer token
→
Gateway
verifies JWT
→
Services
x-user-id + x-user-role
This only works because the services are not publicly accessible Auth and brief services must only accept connections from the gateway — not from the internet. On Railway, this is enforced by keeping them as private services. If someone could directly call http://brief-service:4000 and add fake x-user-role: ADMIN headers, the security would be broken.

authenticate Middleware

The auth service has its own JWT middleware that goes a step further than the gateway — it not only verifies the token but also fetches the full user object from the database and attaches it to req.user. This way controllers can access all user data without repeating the DB lookup.

auth-service/middlewares/authMiddleware.ts
export async function authenticate(req, res, next) {
    try {
        const authHeader = req.headers.authorization
        if (!authHeader || !authHeader.startsWith("Bearer")) {
            res.status(401).json({ message: "No token provided" })
            return
        }
        const token   = authHeader.split(" ")[1]
        const decoded = jwt.verify(token, JWT_SECRET) as { id: string, role: string }

        // Fetch user from DB — confirms account still exists
        const user = await prisma.user.findUnique({ where: { id: decoded.id } })
        if (!user) {
            res.status(401).json({ message: "User not found" })
            return
        }

        // Strip password — never expose it on req.user
        const { password, ...safeUser } = user
        req.user = safeUser    // controllers access this
        next()
    } catch (error) {
        res.status(401).json({ message: "Invalid or expired token" })
    }
}
Why does the auth service check the DB when the gateway already verified the token? The gateway only verifies the JWT signature — it confirms the token was signed with the right secret. But it doesn't check if the user was deleted since the token was issued. The auth service does the DB lookup to catch that case. The brief service doesn't do this because it doesn't have the users database.

Auth Controller

Controllers are functions that handle a specific route. Their job is to: validate the input → call a service function → send the response. They should contain no business logic themselves — that lives in the service layer.

auth-service/src/controllers/authController.ts
export async function register(req: Request, res: Response) {
    try {
        const { firstName, lastName, email, password, phone } = req.body;

        // 1. Check required fields present
        if (!firstName || !lastName || !email || !password) {
            res.status(400).json({ message: "All fields are required" });
            return;
        }

        // 2. Sanitize — trim whitespace, lowercase email
        const cleanFirstName = String(firstName).trim();
        const cleanEmail    = normalizeEmail(String(email));

        // 3. Validate format
        if (!isValidName(cleanFirstName)) {
            res.status(400).json({ message: "Name must be 2-40 letters" });
            return;
        }
        if (!isValidEmail(cleanEmail)) { ... }
        if (!isStrongPassword(cleanPassword)) { ... }

        // 4. Delegate to service layer (the actual logic)
        const result = await initiateRegister(cleanFirstName, cleanLastName, cleanEmail, cleanPassword, cleanPhone)
        res.status(200).json(result)

    } catch (error: any) {
        // Service threw an error (e.g. "Email already in use") → 500
        res.status(500).json({ message: error.message });
    }
}

Auth Service Layer

The service layer contains the actual business logic. Controllers call these functions. This separation means if you wanted to use the same logic in a CLI script or a test, you just import the service function — no HTTP required.

Registration flow — initiateRegister

auth-service/services/authServices.ts
export async function initiateRegister(firstName, lastName, email, password, phone?) {
    // 1. Check email not already registered
    const existing = await prisma.user.findUnique({ where: { email } })
    if (existing) throw new Error("Email already in use")

    // 2. Hash password immediately
    const hashedPassword = await bcrypt.hash(password, 10)

    // 3. Generate 6-digit code, set 10-min expiry
    const code      = generateCode()
    const expiresAt = new Date(Date.now() + 10 * 60 * 1000)

    // 4. Store in VerifyToken table (NOT User table yet)
    // upsert = update if exists, create if not
    // This handles "resend code" — replaces the old pending record
    await prisma.verifyToken.upsert({
        where:  { email },
        update: { code, expiresAt, firstName, lastName, password: hashedPassword, phone },
        create: { email, code, expiresAt, firstName, lastName, password: hashedPassword, phone }
    })

    // 5. Send verification email (fire-and-forget — don't fail registration if email fails)
    try { await sendVerificationEmail(email, code) }
    catch (err) { console.error("Email failed:", err) }

    return { message: "Verification code sent to your email" }
}

Verify code → create user

export async function verifyAndCreateUser(email, code) {
    const record = await prisma.verifyToken.findUnique({ where: { email } })
    if (!record)              throw new Error("No verification pending for this email")
    if (record.code !== code) throw new Error("Invalid code")
    if (record.expiresAt < new Date()) throw new Error("Code has expired")

    // All checks passed — create the actual User record
    const user = await prisma.user.create({
        data: { firstName: record.firstName, lastName: record.lastName,
                email: record.email, password: record.password,
                phone: record.phone, provider: "LOCAL" }
    })

    // Clean up the temporary record
    await prisma.verifyToken.delete({ where: { email } })

    // Sign a JWT and return it with the user
    const token = jwt.sign({ id: user.id, role: user.role }, JWT_SECRET, { expiresIn: "7d" })
    return { user: excludePassword(user), token }
}

Password reset — two-step

// Step 1: requestPasswordReset — user enters email on forgot-password page
export async function requestPasswordReset(email) {
    const user = await prisma.user.findUnique({ where: { email } })

    // Always return success — never reveal if the email exists (security)
    if (!user) return { message: "If this email is registered, a reset link has been sent" }

    // JWT with a special "purpose" field and short 30-min expiry
    const resetToken = jwt.sign(
        { id: user.id, purpose: "reset-password" },
        JWT_SECRET,
        { expiresIn: "30m" }
    )
    const resetLink = `${frontendUrl}/reset-password?token=${resetToken}`
    await sendPasswordResetEmail(email, resetLink)
    return { message: "If this email is registered, a reset link has been sent" }
}

// Step 2: completePasswordReset — user clicks link, enters new password
export async function completePasswordReset(token, newPassword) {
    let decoded: jwt.JwtPayload
    try { decoded = jwt.verify(token, JWT_SECRET) as jwt.JwtPayload }
    catch { throw new Error("Invalid or expired reset token") }

    // Verify the purpose field — a login JWT must not work as a reset token
    if (decoded.purpose !== "reset-password") throw new Error("Invalid token")

    const hashed = await bcrypt.hash(newPassword, 10)
    await prisma.user.update({ where: { id: decoded.id }, data: { password: hashed } })
    return { message: "Password reset successful" }
}

bcrypt — Password Hashing

Passwords are never stored as plain text. bcrypt is a one-way hashing algorithm designed specifically for passwords. Even if the database is leaked, the attacker cannot recover the original passwords.

❌ Never do this
// Storing plain text
await prisma.user.create({
  data: { password: "MyPass123!" }
})
// DB leak → instant compromise
✅ What the project does
// Hash before storing
const hashed = await bcrypt.hash("MyPass123!", 10)
await prisma.user.create({
  data: { password: hashed }
})
// "$2b$10$K5t4XH..." stored in DB
// Hash a password — "10" is the salt rounds (cost factor)
// Higher = slower to hash = slower to brute force
// 10 is the industry standard — takes ~100ms
const hashed = await bcrypt.hash(password, 10)
// Output: "$2b$10$K5t4XHuSGgJSO0AuKoGMTe9LWvNFMmZNr4..." (60 chars always)

// Verify at login — compare plain text to stored hash
// bcrypt.compare is the ONLY way to check — you cannot "decrypt" a hash
const isValid = await bcrypt.compare(plainPassword, user.password)
// true if they match, false otherwise

if (!isValid) throw new Error("Invalid credentials")
Why can't you just hash again and compare? bcrypt deliberately includes a random salt in every hash. The same password hashed twice produces two completely different strings. That's what bcrypt.compare handles internally — it extracts the salt from the stored hash, re-hashes the input with that same salt, then compares.

jsonwebtoken — Signing & Verifying

A JWT (JSON Web Token) is a compact, self-contained token that encodes a payload (like a user's id and role) and signs it with a secret. The signature means anyone with the secret can verify the token was not tampered with — without hitting a database.

// Structure: header.payload.signature
// Example:   eyJhbGciOiJIUzI1NiJ9.eyJpZCI6Ii4uLiIsInJvbGUiOiJBRE1JTiIsImV4cCI6...}

// SIGN — create a token after login/registration
const token = jwt.sign(
    { id: user.id, role: user.role },   // payload — what to encode
    JWT_SECRET,                          // secret — keep this private!
    { expiresIn: "7d" }                // options — token expires in 7 days
)
// Returns a string like "eyJhbGci..."

// VERIFY — check token on every protected request
const decoded = jwt.verify(token, JWT_SECRET) as { id: string, role: string }
// Returns the payload object: { id: "...", role: "ADMIN", iat: 1234, exp: 1234 }
// Throws if: signature invalid, token expired, token malformed

// The reset-password token has an extra "purpose" field
const resetToken = jwt.sign(
    { id: user.id, purpose: "reset-password" },
    JWT_SECRET,
    { expiresIn: "30m" }  // short-lived — only 30 minutes
)
Token typePayloadExpiryUsed for
Auth token{ id, role }7 daysEvery authenticated request
Reset token{ id, purpose: "reset-password" }30 minPassword reset link in email
JWT_SECRET must be kept secret and be long If someone knows your JWT_SECRET, they can forge tokens and impersonate any user including admins. It should be a long random string (32+ characters), stored only in environment variables — never in code or git.

passport.use(
    new GoogleStrategy(
        {
            clientID:     process.env.GOOGLE_CLIENT_ID,
            clientSecret: process.env.GOOGLE_CLIENT_SECRET,
            callbackURL:  process.env.GOOGLE_CALLBACK_URL,
            // callbackURL = "https://gateway.railway.app/auth/google/callback"

        async (_accessToken, _refreshToken, profile, done) => {
            try {
                // profile contains: id, emails[0].value, name.givenName, etc.
                const result = await findOrCreateGoogleUser(profile)
                done(null, result)  // success — result attached to req.user
            } catch(error) {
                done(error, undefined) // failure
            }
        }
    )
)
auth-service/services/authServices.ts — findOrCreateGoogleUser
export async function findOrCreateGoogleUser(profile) {
    const { id: googleId, emails, name } = profile
    const email = emails[0].value

    // First try to find by googleId (returning Google user)
    let user = await prisma.user.findUnique({ where: { googleId } })

    if (!user) {
        // upsert by email: if email exists (LOCAL account), link Google to it
        // if email doesn't exist, create a new user entirely
        user = await prisma.user.upsert({
            where:  { email },
            update: { googleId, provider: "GOOGLE" },
            create: {
                firstName: name.givenName, lastName: name.familyName,
                email, googleId, provider: "GOOGLE"
            }
        })
    }

    // Sign a JWT just like regular login
    const token = jwt.sign({ id: user.id, role: user.role }, JWT_SECRET, { expiresIn: "7d" })
    return { user: excludePassword(user), token }
}
auth-service/src/controllers/authController.ts — googleCallback
export async function googleCallback(req, res) {
    // req.user was set by the Google strategy (via done(null, result))
    const { token } = req.user as any
    const frontendUrl = process.env.FRONTEND_URL || "http://localhost:5173"
    // Redirect back to React app with the token in the URL
    // App.tsx reads this with useSearchParams → saves to localStorage → cleans URL
    res.redirect(`${frontendUrl}?token=${token}`)
}

Input Validation Utilities

The auth service uses custom regex-based validation functions in utils/validation.ts. These run in the controller before any DB call — bad input is rejected early.

auth-service/utils/validation.ts
// Email: must have local@domain.tld structure
const EMAIL_REGEX = /^[^\s@]+@[^\s@]+\.[^\s@]{2,}$/i
export function isValidEmail(email) { return EMAIL_REGEX.test(email.trim()) }

// Name: letters, spaces, hyphens, apostrophes — 2 to 40 chars
const NAME_REGEX = /^[A-Za-z][A-Za-z\s'-]{1,39}$/
export function isValidName(name) { return NAME_REGEX.test(name.trim()) }

// Password strength: 8+ chars, uppercase, lowercase, digit, special char
export function isStrongPassword(password) {
    if (password.length < 8)             return false
    if (!/[A-Z]/.test(password))        return false
    if (!/[a-z]/.test(password))        return false
    if (!/\d/.test(password))           return false
    if (!/[!@#$%^&*(),.?":{}|<>]/.test(password)) return false
    return true
}

// Phone: 9-15 digits (strips non-digits first)
export function isValidPhone(phone) {
    const normalized = phone.replace(/\D/g, "")
    return normalized.length >= 9 && normalized.length <= 15
}

// Email normalization — lowercase and strip whitespace
export function normalizeEmail(email) { return email.trim().toLowerCase() }

requireAuth Middleware

The brief service does not have a database of users and cannot verify JWTs. Instead it trusts the headers injected by the gateway. If x-user-id and x-user-role are present, the request is authenticated. If they're missing, the request didn't come through the gateway and must be rejected.

brief-service/src/middlewares/auth.middleware.ts
export function requireAuth(req, res, next) {
    const userId = req.header("x-user-id")
    const role   = req.header("x-user-role") as UserRole | undefined

    if (!userId || !role) {
        res.status(401).json({ message: "Unauthorized" })
        return
    }

    // Attach auth context to req.user for all downstream handlers
    req.user = { userId, role }
    next()
}

Note also the TypeScript module augmentation in this file — it extends the Express Request type to include user:

// Without this, TypeScript would complain that req.user doesn't exist
declare global {
    namespace Express {
        interface Request {
            user?: AuthContext   // { userId: string, role: UserRole }
        }
    }
}

Zod — Schema Validation

The brief service uses Zod instead of manual validation. Zod lets you declare the shape of your data as a schema, then call .parse() — it validates and transforms the input in one step, throwing a ZodError with detailed messages if anything is wrong.

brief-service/src/schemas/brief.schema.ts
import { z } from "zod"

// Enum schema — only these values are accepted
export const BriefStatusEnum = z.enum(["PENDING", "ACCEPTED", "REFUSED", "IN_PROGRESS", "COMPLETED"])

export const createBriefSchema = z.object({
    title:       z.string().min(1, "Title is required"),
    projectType: ProjectTypeEnum,
    description: z.string().min(1, "Description is required"),
    features:    z.array(z.string()).min(1, "At least one feature"),
    budgetRange: z.string().min(1),
    deadline:    z.string()
                  .refine(str => !isNaN(Date.parse(str)), "Invalid date")
                  .transform(str => new Date(str)),  // string → Date object
    attachments: z.array(z.string()).optional().default([]),
})

export const assignBriefSchema = z.object({
    employeeIds: z.array(z.string().uuid("Invalid employee ID"))
                  .min(1, "At least one employee"),
})
// In a controller — parse throws ZodError on invalid input
const body = createBriefSchema.parse(req.body)
// body is now fully typed and validated
// body.deadline is a Date object (not a string anymore)

// Catch the ZodError in the controller and return helpful messages
try {
    const body = createBriefSchema.parse(req.body)
    ...
} catch (error) {
    if (error instanceof z.ZodError) {
        return res.status(400).json({ errors: error.issues })
        // [{ path: ["title"], message: "Title is required" }, ...]
    }
}
Zod methodWhat it validates
z.string().min(1)Non-empty string
z.string().uuid()Valid UUID format
z.enum([...])Value must be one of the listed options
z.array(z.string())Array of strings
.optional()Field is not required
.default([])Use this if field is missing
.refine(fn, msg)Custom validation function
.transform(fn)Transform after validation (string → Date)

Brief Controller — Role Guards & Patterns

The brief controller's handlers follow a consistent pattern: check the role, validate the body with Zod, call the service, optionally send a notification, respond.

brief-service/src/controllers/brief.controller.ts
// Helper to get auth context from req.user (set by requireAuth middleware)
const getAuth = (req: Request): briefService.AuthContext => req.user as any

// ADMIN ONLY — update brief status (accept/refuse)
export const updateBriefStatusHandler = async (req, res) => {
    try {
        const auth    = getAuth(req)
        if (auth.role !== "ADMIN") return res.status(403).json({ error: "Forbidden" })

        const briefId = req.params.briefId
        const body    = updateBriefStatusSchema.parse(req.body)  // validate

        const updated = await briefService.updateBriefStatus(briefId, body.status, auth.userId, body.reason)

        // Fire-and-forget notification — don't fail the request if email breaks
        try {
            const usersMap = await fetchUsersFromAuthService([updated.clientId])
            const client   = usersMap.get(updated.clientId)
            if (client?.email) {
                await sendBriefStatusEmail(client.email, `${client.firstName} ${client.lastName}`, updated.title, updated.status, updated.statusReason)
            }
        } catch (err) { console.error("Email error:", err) }

        res.status(200).json(updated)
    } catch (error: any) {
        if (error instanceof z.ZodError) return res.status(400).json({ errors: error.issues })
        res.status(400).json({ error: error.message })
    }
}

Brief Service Layer — Business Logic

The service layer enforces all business rules. The controller just calls these — it doesn't know anything about permissions or state machine logic.

brief-service/src/services/briefService.ts
// Clients can only edit briefs in PENDING status
const EDITABLE_BY_CLIENT: BriefStatus[] = ["PENDING"]

export async function updateClientBrief(clientId, briefId, input) {
    const brief = await prisma.brief.findUnique({ where: { id: briefId } })
    if (!brief)                                       throw new Error("Brief not found")
    if (brief.clientId !== clientId)                  throw new Error("Forbidden")
    if (!EDITABLE_BY_CLIENT.includes(brief.status))   throw new Error("Brief cannot be edited in current status")
    return prisma.brief.update({ where: { id: briefId }, data: input })
}

// Employees can only set status to IN_PROGRESS or COMPLETED
const EMPLOYEE_ALLOWED_STATUS: BriefStatus[] = ["IN_PROGRESS", "COMPLETED"]

export async function updateEmployeeBriefStatus(briefId, employeeId, status) {
    if (!EMPLOYEE_ALLOWED_STATUS.includes(status)) throw new Error("Status not allowed for employee")

    const brief = await prisma.brief.findUnique({ where: { id: briefId } })
    if (!brief)                                       throw new Error("Brief not found")
    if (!brief.assignedToIds.includes(employeeId))    throw new Error("Forbidden")

    return prisma.brief.update({ where: { id: briefId },
        data: { status, statusUpdatedById: employeeId, statusUpdatedAt: new Date() }
    })
}

// assignBrief also sets status to IN_PROGRESS automatically
export async function assignBrief(briefId, employeeIds, adminId) {
    return prisma.brief.update({
        where: { id: briefId },
        data: {
            assignedToIds:      employeeIds,
            assignedById:       adminId,
            assignedAt:         new Date(),
            status:             "IN_PROGRESS",  // auto-progress on assign
            statusUpdatedById:  adminId,
            statusUpdatedAt:    new Date(),
        }
    })
}

// getBriefById enforces per-role access
export async function getBriefById(briefId, requester) {
    const brief = await prisma.brief.findUnique({ where: { id: briefId } })
    if (!brief) throw new Error("Brief not found")

    const isClientOwner    = requester.role === "CLIENT"   && brief.clientId === requester.userId
    const isAssignedEmployee = requester.role === "EMPLOYEE" && brief.assignedToIds.includes(requester.userId)
    const isAdmin          = requester.role === "ADMIN"

    if (!isClientOwner && !isAssignedEmployee && !isAdmin) {
        throw new Error("Forbidden")
    }
    return brief
}

Service-to-Service Communication

The brief service needs user details (names, emails) to send notification emails — but it has no users database. It calls the auth service through the gateway using plain HTTP with Axios.

brief-service/src/utils/authClient.ts
export async function fetchUsersFromAuthService(userIds: string[]): Promise<Map<string, UserSnapshot>> {
    // deduplicate IDs
    const uniqueIds = [...new Set(userIds.filter(Boolean))]
    if (!uniqueIds.length) return new Map()

    const gatewayUrl = process.env.GATEWAY_URL || "http://localhost:8080"

    const response = await axios.get<UserSnapshot[]>(
        `${gatewayUrl}/api/users/internal`,
        { params: { ids: uniqueIds.join(",") } }
        // GET /api/users/internal?ids=id1,id2,id3
    )

    // Convert array to Map for O(1) lookup by ID
    return new Map(response.data.map(user => [user.id, user]))
}

// Usage in the controller — get one or many users
const usersMap = await fetchUsersFromAuthService([brief.clientId, ...brief.assignedToIds])
const client   = usersMap.get(brief.clientId)       // UserSnapshot | undefined
const employees = brief.assignedToIds.map(id => usersMap.get(id)).filter(Boolean)

The gateway then proxies this to the auth service's internal endpoint which has no auth check:

The Middleware Chain

Middleware is the most important concept in Express. Every app.use() or route handler is middleware. Each one receives (req, res, next) and either:

  • Calls next() to pass control to the next middleware
  • Sends a response (res.json(...)) which ends the chain
// Order matters — they run top to bottom
app.use(cors())               // 1. CORS headers added
app.use(express.json())       // 2. Body parsed into req.body
app.use(logger)               // 3. Log the request
app.use(authenticate)         // 4. Verify JWT — calls next() or sends 401
app.use("/auth", authRouter)   // 5. Route to correct handler

// Inside a Router, middleware also chains:
router.get("/me", authenticate, getMe)
//              ↑ runs first    ↑ runs if authenticate calls next()
Why return after res.json()? In TypeScript with Express, after sending a response you should return to prevent the function from continuing and accidentally calling next() or sending another response. The project does this consistently: res.status(400).json(...); return;

Module Augmentation — declare global

Express's Request type does not have a user property by default. TypeScript would give an error if you tried to access req.user. Module augmentation lets you extend an existing type from a library without modifying the library itself:

brief-service/src/middlewares/auth.middleware.ts
// Tell TypeScript: the Express Request interface now also has this field
declare global {
    namespace Express {
        interface Request {
            user?: AuthContext   // { userId: string; role: UserRole }
        }
    }
}

// Now this works in every file without TypeScript complaining
req.user = { userId, role }   // set it in middleware
const auth = req.user           // read it in controllers

The auth service does the same in its middleware — it sets req.user to the full Prisma User object (minus password). Both services augment the same Express.Request type, each with the shape relevant to their service.

Error Handling Pattern

Every controller in Smart Brief uses the same try/catch structure. Service functions throw plain Error objects with meaningful messages. The controller catches them and decides the right HTTP status:

// Service function throws with a meaningful message
export async function deleteBrief(briefId, requester) {
    const brief = await prisma.brief.findUnique({ where: { id: briefId } })
    if (!brief) throw new Error("Brief not found")    // controller maps to 404

    const allowed = requester.role === "ADMIN"
        || (requester.role === "CLIENT" && brief.clientId === requester.userId && brief.status === "PENDING")
    if (!allowed) throw new Error("Forbidden")            // controller maps to 403

    return prisma.brief.delete({ where: { id: briefId } })
}

// Controller inspects the error message to choose the right status
export const deleteBriefHandler = async (req, res) => {
    try {
        const auth    = getAuth(req)
        const briefId = req.params.briefId
        await briefService.deleteBrief(briefId, auth)
        res.status(204).send()
    } catch (error: any) {
        const status =
            error.message === "Forbidden"       ? 403 :
            error.message === "Brief not found" ? 404 : 500
        res.status(status).json({ error: error.message })
    }
}
