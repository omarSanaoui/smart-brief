Docker

Complete guide — from why it exists to Dockerizing the entire Smart Brief stack.

Why Docker Exists

Imagine this scenario:

"It works on my machine."

You build an app on Windows with Node 20. Your teammate runs it on macOS with Node 18. It crashes. The production server runs Ubuntu with Node 16. It crashes differently. The database version is different. The environment variables are in different places. The file paths use different separators.

This is the problem Docker solves: making an application run the same way everywhere — your laptop, your teammate's laptop, CI/CD, production.

What Docker does

Docker packages your app and everything it needs to run into a single unit called a container:

  • The exact Node.js version
  • All npm packages
  • The OS libraries
  • The configuration
  • The start command

Anyone with Docker installed can run your container and it will behave exactly as it did on your machine.

What is Docker?

Docker is a tool for building and running containers. A container is like a small, isolated computer running inside your computer. It has:

  • Its own filesystem
  • Its own processes
  • Its own network interface
  • Its own environment variables

But unlike a full virtual machine, a container shares the host OS kernel, making it lightweight and fast to start (seconds instead of minutes).

Key distinction An image is the blueprint (read-only). A container is a running instance of an image. You can run many containers from the same image.

Image = class definition. Container = instance of that class.

Image

A Docker image is a read-only template used to create containers. It's a snapshot of a filesystem with your app and all its dependencies baked in.

Images are built from a Dockerfile — a text file that lists instructions for building the image layer by layer.

Images are identified by a name and tag:

# Format: name:tag
node:20-alpine        # official Node.js 20 on Alpine Linux
node:20               # Node.js 20 on Debian
postgres:15           # official PostgreSQL 15
nginx:latest          # latest nginx
smart-brief/auth:1.0  # your own image

Images are stored in registries. The default is Docker Hub (hub.docker.com). You can also use GitHub Container Registry, AWS ECR, etc.

Container

A container is a running instance of an image. When you run an image, Docker creates a container — an isolated process with its own filesystem, network, and environment. 

# Run one container from node:20 image
docker run node:20 node --version   # prints v20.x.x then exits

# Run multiple containers from the same image simultaneously
docker run -d node:20 ...  # container 1
docker run -d node:20 ...  # container 2 — totally independent

Containers are ephemeral by default — when a container stops, any data written inside it is lost. For persistent data (databases), you use volumes.

Dockerfile

A Dockerfile is a text file (no extension) that defines how to build an image. It lives at the root of your project. 

# Every line is an instruction that creates a new layer

FROM    node:20-alpine        # start from this base image
WORKDIR /app                  # set working directory inside container
COPY    package*.json ./      # copy package files
RUN     npm ci                # install dependencies
COPY    . .                   # copy rest of source code
EXPOSE  3000                  # document which port the app uses
CMD     ["node", "dist/index.js"]  # command to start the app

Layers

Every instruction in a Dockerfile creates a layer. Layers are cached — if nothing changed in a layer, Docker reuses the cache and skips re-running it.

FROM node:20-alpine
base OS + Node.js runtime
COPY package*.json
RUN npm ci
node_modules — slow, but cached
COPY . .
your source code — changes often
CMD ["node", "dist/index.js"]
start command

The most important insight about layers: put things that change rarely near the top, things that change often near the bottom.

This is why you always copy package.json first, install deps, then copy source code:

  • package.json changes rarely → npm install is cached
  • Your source code changes every build → only this layer rebuilds

If you did COPY . . then RUN npm install, every single code change would re-run npm install — very slow.

CMD vs ENTRYPOINT CMD is the default command — you can override it when running the container: docker run myapp node other.js. ENTRYPOINT always runs and CMD becomes its arguments. For simple Node apps, use CMD.

Node.js Dockerfile

The correct Dockerfile for a TypeScript Node.js service:

# ──────────────────────────────────────────────
# 1. Base image — Alpine Linux is tiny (~5MB)
# ──────────────────────────────────────────────
FROM node:20-alpine

# 2. Set working directory — all commands run from here
WORKDIR /app

# 3. Copy ONLY package files first
#    This layer is cached as long as package.json doesn't change
COPY package.json package-lock.json ./

# 4. Install dependencies (clean install, respects lockfile)
RUN npm ci

# 5. Copy Prisma schema before generating client
COPY prisma ./prisma

# 6. Generate Prisma client
RUN npx prisma generate

# 7. Copy rest of source code (changes most often — last step)
COPY . .

# 8. Build TypeScript to JavaScript
RUN npm run build

# 9. Tell Docker which port this service uses
EXPOSE 3000

# 10. Run migrations then start the server
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/index.js"]

Which base image to use?

ImageSizeUse case
node:20~1.1 GBFull Debian — maximum compatibility
node:20-slim~250 MBDebian without build tools — good default

Alpine and native modules Some npm packages with native C bindings (like bcrypt — not bcryptjs) can fail on Alpine because it uses musl libc instead of glibc. Smart Brief uses bcryptjs (pure JS) which works fine on Alpine.

.dockerignore

Like .gitignore but for Docker. Files listed here are excluded from the build context — they won't be sent to Docker or copied into the image. 

# .dockerignore
node_modules        # use npm ci inside Docker, not your local node_modules
dist                # TypeScript output — will be rebuilt inside Docker
.env                # secrets — inject at runtime, never bake into image
.env.*
*.log
.git
.gitignore
README.md
docs/
src/generated/      # Prisma generated files — regenerated during build
Never COPY .env into a Docker image Images can be pushed to registries, shared with teammates, or inspected. A .env baked into an image exposes your secrets to anyone who has the image. Pass env vars at runtime using -e flags or docker-compose.yml.

Multi-Stage Builds

A multi-stage build uses multiple FROM statements in one Dockerfile. You build in one stage and only copy the output into the final image — keeping the final image small. 

# Stage 1: builder — has TypeScript compiler, devDependencies, etc.
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN  npm ci                   # installs ALL deps including devDeps
COPY prisma ./prisma
RUN  npx prisma generate
COPY . .
RUN  npm run build            # compiles TypeScript → dist/

# Stage 2: production — only what's needed to RUN the app
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN  npm ci --omit=dev       # only production deps
COPY prisma ./prisma
RUN  npx prisma generate
COPY --from=builder /app/dist ./dist   # only copy compiled output
EXPOSE 3000
CMD  ["sh", "-c", "npx prisma migrate deploy && node dist/index.js"]

Result: the final image contains no TypeScript compiler, no devDependencies, no source code — just the compiled JS and runtime deps. Much smaller and more secure.

docker build

# Basic build — creates image tagged with a name
docker build -t smart-brief-auth .

# With version tag
docker build -t smart-brief-auth:1.0.0 .

# Build from a specific Dockerfile (not named "Dockerfile")
docker build -f Dockerfile.prod -t smart-brief-auth .

# Build a specific service's Dockerfile from the monorepo root
docker build -f services/auth-service/Dockerfile -t auth-service .

# List all locally built images
docker images
The dot at the end The . at the end of docker build is the build context — the folder Docker sends to the build engine. All COPY instructions are relative to this directory. Usually it's . (current folder), but you can specify another path.

docker run

# Basic run — runs interactively, blocks terminal
docker run smart-brief-auth

# Run in detached mode (background)
docker run -d smart-brief-auth

# Run with a name (easier to reference)
docker run -d --name auth smart-brief-auth

# Map host port to container port  -p HOST:CONTAINER
docker run -d -p 3000:3000 smart-brief-auth
#               ↑    ↑
#        your laptop  container's internal port
# Now http://localhost:3000 reaches the container

# Pass environment variables
docker run -d -p 3000:3000 \
  -e DATABASE_URL=postgresql://... \
  -e JWT_SECRET=mysecret \
  smart-brief-auth

# Mount a local folder into the container (volume)
docker run -v /local/path:/container/path smart-brief-auth

# Auto-remove when stopped
docker run --rm smart-brief-auth

Port mapping explained

# Your app inside the container listens on port 3000
# But that port is internal — only reachable inside the container
# -p 3000:3000 maps HOST:CONTAINER
# -p 8080:3000 → your machine's :8080 maps to container's :3000

docker run -p 8080:3000 smart-brief-auth
# Now http://localhost:8080 → container's :3000

Why Docker Compose?

Running multiple containers manually with docker run gets messy fast. For Smart Brief you'd need to:

  • Run docker run for PostgreSQL (auth DB)
  • Run docker run for PostgreSQL (brief DB)
  • Run docker run for auth-service (with 10 -e flags)
  • Run docker run for brief-service (with 10 -e flags)
  • Run docker run for gateway (with 10 -e flags)
  • Make sure they can all talk to each other
  • Run all that in the right order

Docker Compose solves this. It lets you define all your services in one docker-compose.yml file and start everything with one command: docker compose up.

docker-compose.yml

A minimal example with two services sharing a network:

version: "3.9"

services:

  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER:     myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB:       mydb
    volumes:
      - postgres_data:/var/lib/postgresql/data  # persist data
    ports:
      - "5432:5432"  # expose to host for local psql

  api:
    build: .             # build from Dockerfile in current dir
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://myuser:mypassword@postgres:5432/mydb
      #                                              ↑
      #                                 service name — not localhost!
    depends_on:
      - postgres

volumes:
  postgres_data:  # declare named volume at bottom
Inside Docker Compose, services talk to each other by service name, not localhost When the api container connects to the database, it uses postgres (the service name) instead of localhost. Docker Compose creates a private network and registers each service name as a DNS entry.

# Start all services (builds if needed)

# Inside the gateway container, reach auth-service:
http://auth-service:3000   ← works!
http://localhost:3000      ← does NOT work (localhost = the gateway container itself)

# From your host machine (browser), reach the gateway:
http://localhost:8080      ← works (because of -p 8080:8080)
http://gateway:8080        ← does NOT work (your machine isn't on the Docker network)

This is why environment variables change when you use Docker Compose:

# Without Docker (everything on localhost):
AUTH_SERVICE_URL=http://localhost:3000
BRIEF_SERVICE_URL=http://localhost:4000
DATABASE_URL=postgresql://user:pass@localhost:5432/db

# With Docker Compose (use service names):
AUTH_SERVICE_URL=http://auth-service:3000
BRIEF_SERVICE_URL=http://brief-service:4000
DATABASE_URL=postgresql://user:pass@auth-db:5432/auth_db

Volumes & Persistence

Containers are ephemeral — data written inside is lost when the container is removed. For databases, you need a volume to persist data outside the container.

Named volumes (recommended)

services:
  auth-db:
    image: postgres:15-alpine
    volumes:
      - auth_postgres_data:/var/lib/postgresql/data
      #     ↑ volume name       ↑ path inside container where postgres stores files

volumes:
  auth_postgres_data:  # declare the named volume — Docker manages it

Bind mounts — map a local folder

# Useful for development — changes to local files appear inside container
services:
  auth-service:
    volumes:
      - ./services/auth-service:/app   # local folder → container /app
      - /app/node_modules              # don't override node_modules
Hot reload in development Use a bind mount + tsx watch for hot reload in containers. When you save a file locally, the container sees the change and restarts automatically — no need to rebuild the image every time.

Env Files in Compose

# Option 1: Inline in docker-compose.yml (visible to anyone)
environment:
  DATABASE_URL: postgresql://user:pass@db:5432/mydb

# Option 2: env_file — read from a file (add file to .gitignore)
env_file:
  - .env                    # reads all KEY=VALUE lines from .env
  - .env.local              # overrides with local values

# Option 3: Reference variables from host environment
environment:
  DATABASE_URL: ${DATABASE_URL}   # reads from your shell's environment

docker-compose.yml also reads a top-level .env file automatically:

# .env at the same level as docker-compose.yml
POSTGRES_PASSWORD=supersecret
JWT_SECRET=myjwtsecret

# docker-compose.yml can reference it with ${VARIABLE}
environment:
  POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
  JWT_SECRET: ${JWT_SECRET}

depends_on & Health Checks

depends_on tells Compose to start services in order. But by default it only waits for the container to start — not for the service inside to be ready. A PostgreSQL container can be running but the database not yet accepting connections.

depends_on with condition (correct approach)

services:

  auth-db:
    image: postgres:15-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB"]
      interval: 5s
      timeout: 5s
      retries: 5

  auth-service:
    build: ./services/auth-service
    depends_on:
      auth-db:
        condition: service_healthy  # waits for DB to be truly ready

Now auth-service won't start until PostgreSQL passes the health check.

Full docker-compose.yml for Smart Brief

Place this at the root of the project (smart-brief/docker-compose.yml):

version: "3.9"

services:

  # ─── Auth Database ───────────────────────────────────────────
  auth-db:
    image: postgres:15-alpine
    container_name: smart-brief-auth-db
    environment:
      POSTGRES_USER:     authuser
      POSTGRES_PASSWORD: authpassword
      POSTGRES_DB:       authdb
    volumes:
      - auth_db_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"    # expose for local psql access
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U authuser -d authdb"]
      interval: 5s
      timeout: 5s
      retries: 5

  # ─── Brief Database ──────────────────────────────────────────
  brief-db:
    image: postgres:15-alpine
    container_name: smart-brief-brief-db
    environment:
      POSTGRES_USER:     briefuser
      POSTGRES_PASSWORD: briefpassword
      POSTGRES_DB:       briefdb
    volumes:
      - brief_db_data:/var/lib/postgresql/data
    ports:
      - "5433:5432"    # different host port to avoid conflict
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U briefuser -d briefdb"]
      interval: 5s
      timeout: 5s
      retries: 5

  # ─── Auth Service ────────────────────────────────────────────
  auth-service:
    build:
      context: ./services/auth-service
      dockerfile: Dockerfile
    container_name: smart-brief-auth
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL:       postgresql://authuser:authpassword@auth-db:5432/authdb
      JWT_SECRET:         ${JWT_SECRET}
      GOOGLE_CLIENT_ID:   ${GOOGLE_CLIENT_ID}
      GOOGLE_CLIENT_SECRET: ${GOOGLE_CLIENT_SECRET}
      GOOGLE_CALLBACK_URL: ${GOOGLE_CALLBACK_URL}
      FRONTEND_URL:       http://localhost:5173
    depends_on:
      auth-db:
        condition: service_healthy
    restart: unless-stopped

  # ─── Brief Service ───────────────────────────────────────────
  brief-service:
    build:
      context: ./services/brief-service
      dockerfile: Dockerfile
    container_name: smart-brief-brief
    ports:
      - "4000:4000"
    environment:
      DATABASE_URL: postgresql://briefuser:briefpassword@brief-db:5432/briefdb
      GATEWAY_URL:  http://gateway:8080
      ADMIN_EMAIL:  ${ADMIN_EMAIL}
    depends_on:
      brief-db:
        condition: service_healthy
    restart: unless-stopped

  # ─── Gateway ─────────────────────────────────────────────────
  gateway:
    build:
      context: ./services/gateway
      dockerfile: Dockerfile
    container_name: smart-brief-gateway
    ports:
      - "8080:8080"     # the only port exposed to the browser
    environment:
      JWT_SECRET:       ${JWT_SECRET}
      AUTH_SERVICE_URL: http://auth-service:3000
      BRIEF_SERVICE_URL: http://brief-service:4000
      FRONTEND_URL:     http://localhost:5173
    depends_on:
      - auth-service
      - brief-service
    restart: unless-stopped

# ─── Named volumes ─────────────────────────────────────────────
volumes:
  auth_db_data:
  brief_db_data:

When to Use Docker

SituationUse Docker?Why
Running PostgreSQL locally✅ YesNo system install needed, any version, easy to reset
Sharing a project with teammates✅ Yes"Works on my machine" disappears
CI/CD pipelines✅ YesReproducible builds, test in same env as production
Running microservices locally✅ YesOne command to spin up all 5 services

Docker vs Railway/Vercel

Docker (self-managed)

  • Full control of everything
  • Run on any VPS (DigitalOcean, EC2, Hetzner)
  • Usually cheaper at scale
  • You manage scaling, networking, TLS
  • More complex setup and maintenance
# On a VPS:
docker compose pull
docker compose up -d

Railway/Vercel/Render (PaaS)

  • Zero server management
  • Auto-deploy from git push
  • TLS, scaling, logging built-in
  • Can use Dockerfile OR auto-detect
  • More expensive at scale but zero ops cost
# Railway auto-detects a Dockerfile
# and builds it on every git push
Smart Brief on Railway — does it use Docker? Railway can automatically detect a Dockerfile and use it. If you add Dockerfiles to your services, Railway will use them. If not, Railway uses its own Nixpacks build system (which detects Node.js automatically). Either works — adding Dockerfiles gives you more control over the exact environment.

Docker in Production

Security best practices

# Don't run as root — use the built-in 'node' user
FROM node:20-alpine
WORKDIR /app
COPY --chown=node:node package*.json ./
RUN npm ci
COPY --chown=node:node . .
RUN npm run build

USER node   ← switch to non-root before CMD
CMD ["node", "dist/index.js"]
# Never store secrets in the image
ENV JWT_SECRET=hardcoded   ← NEVER do this

# Instead, pass at runtime:
docker run -e JWT_SECRET=$JWT_SECRET myimage
# Or use Docker secrets / Compose secrets for production

Useful production flags

restart: unless-stopped   # auto-restart if container crashes
restart: always           # always restart, even on docker daemon restart
restart: on-failure       # only restart on non-zero exit code

Resource limits

services:
  auth-service:
    deploy:
      resources:
        limits:
          cpus: "0.5"     # max 50% of one CPU core
          memory: 512M    # max 512 MB RAM

# Run Prisma Studio inside the container
docker compose exec auth-service npx prisma studio

# Run a migration manually
docker compose exec auth-service npx prisma migrate deploy

# Check disk usage by images and containers
docker system df

# Clean up everything unused (images, containers, networks, caches)
docker system prune -a
