Email & PDF Generation

Sending transactional emails with Resend and generating PDF documents with PDFKit — the complete guide with every Smart Brief example.

Why Resend?

To send emails from Node.js you need an email delivery service. You can't just call sendEmail() — emails sent directly from a server almost always land in spam because they're not coming from a trusted sender with proper DNS records (SPF, DKIM, DMARC).

Nodemailer + SMTP

  • Send through Gmail, Outlook, or any SMTP server
  • Free with personal Gmail account
  • Gmail limits: 500 emails/day
  • Need to configure SMTP host, port, auth
  • Often lands in spam for cold recipients
  • Google forces "App Passwords" (confusing)

Resend

  • Purpose-built developer email API
  • Free tier: 3,000 emails/month, 100/day
  • One function call to send
  • Handles deliverability (SPF, DKIM)
  • Dashboard to see sent emails, opens, bounces
  • React Email integration for templates

Smart Brief uses Resend. The free tier is enough for a real project.

Installation & Setup

# Install the Resend SDK
npm install resend

Get your API key

  1. Go to resend.com and create a free account
  2. In the dashboard → API Keys → Create API Key
  3. Copy the key (starts with re_...)
  4. Add to your .env file:
RESEND_API_KEY=re_yourKeyHere_abc123

The FROM address

By default (free plan, no domain), Resend gives you a sandbox sender: onboarding@resend.dev — this only works for sending to your own verified email. 

// This is what Smart Brief uses on the free plan
const FROM = 'Smart Brief <onboarding@resend.dev>'

To send to anyone, you need to add and verify your own domain in the Resend dashboard, then use noreply@yourdomain.com as the FROM.

Free plan limitation With onboarding@resend.dev, you can only send to email addresses you've added and verified in your Resend account. For a real production app, add your domain and use your own FROM address.

Lazy Init Pattern

Smart Brief creates the Resend client lazily — only when a function is called — instead of at module load time:

// ❌ Created at module load — crashes if RESEND_API_KEY is missing
const resend = new Resend(process.env.RESEND_API_KEY)

// ✅ Created on first use — safe even if env var is missing in some envs
const getResend = () => new Resend(process.env.RESEND_API_KEY)

// Usage: call getResend() inside each send function
await getResend().emails.send({ ... })
Why lazy init matters If you create new Resend(undefined) at module load, the Resend SDK throws immediately when the module is imported — even in environments where you're not actually sending emails (like tests). Lazy init defers that crash to the moment you actually try to send.

HTML Emails

Email clients (Gmail, Outlook, Apple Mail) are very limited in what CSS they support. Modern CSS like flexbox, grid, var(), or external stylesheets often don't work. The industry standard is to write emails like it's 1999 — HTML tables and inline styles.

What doesn't work in emails CSS classes (class="btn" with a stylesheet), CSS variables (var(--color)), flexbox, grid, position: absolute, web fonts (sometimes), <link> stylesheets, JavaScript (always blocked).
What DOES work Inline styles (style="color:red;"), HTML tables for layout, <td> for columns, basic CSS like color, font-size, background-color, padding, border-radius.

The Layout Wrapper Pattern

Smart Brief defines a layout(content) function that wraps any content in the branded shell. Every email function just writes the inner content and calls layout(). This avoids repeating the header/footer HTML in every email.

                     padding:24px 36px;border-bottom:3px solid #414CC4;">

Error Handling

Email sending is a side effect — it should never make the main operation fail. If you accept a brief and the email crashes, the brief should still be accepted. Always wrap email calls in a try/catch that doesn't re-throw: 

// In brief.controller.ts — this is the pattern everywhere
const updated = await briefService.updateBriefStatus(briefId, status, adminId)
// ↑ The real work — must succeed

try {
  const usersMap = await fetchUsersFromAuthService([updated.clientId])
  const client = usersMap.get(updated.clientId)
  if (client?.email) {
    await sendBriefStatusEmail(client.email, ...)
  }
} catch (err) {
  console.error("Could not send email:", err)
  // ← do NOT throw. Brief update already succeeded.
}

res.status(200).json(updated)  // always respond

Also guard against missing API key to avoid crashes in test/dev environments:

export async function sendBriefStatusEmail(...) {
  if (!process.env.RESEND_API_KEY) return  // silent skip
  // proceed with send...
}

What is PDFKit?

PDFKit is a Node.js library for creating PDF documents programmatically. You describe what to draw — text, shapes, lines, images — using an API, and PDFKit produces a binary PDF file.

Think of it like a canvas: you pick coordinates and draw things. PDFKit also has a text flow mode where it handles line wrapping and cursor movement automatically.

Installation

npm install pdfkit
npm install --save-dev @types/pdfkit   # TypeScript types

How PDFKit Works

PDFKit is a stream. You create a document, add content to it, then call doc.end() to finalize it. The data comes out as a series of chunks through the 'data' event.

new PDFDocument()
→
add text/shapes
→
doc.end()
→
'data' events
chunks of bytes
→
Buffer.concat()
= complete PDF

Promise + Buffer Pattern

PDFKit is event-based, but we want to use async/await. The solution is to wrap the stream events in a Promise and collect all chunks into a Buffer: 

import PDFDocument from "pdfkit"

async function generatePDF(): Promise<Buffer> {
  return new Promise((resolve, reject) => {
    const doc = new PDFDocument({ margin: 50, size: "A4" })
    const chunks: Buffer[] = []

    // Collect every chunk of PDF data
    doc.on("data", (chunk) => chunks.push(Buffer.isBuffer(chunk) ? chunk : Buffer.from(chunk)))

    // When done — merge all chunks and resolve
    doc.on("end", () => resolve(Buffer.concat(chunks)))

    // If error — reject the promise
    doc.on("error", reject)

    // ─── Draw your content here ───────────────────────
    doc.fontSize(24).text("Hello World")
    doc.moveDown()
    doc.fontSize(14).text("This is a PDF")
    // ──────────────────────────────────────────────────

    doc.end()  // ← must call this to trigger 'end' event
  })
}

// Usage:
const pdfBuffer = await generatePDF()
// pdfBuffer is a Buffer containing the complete PDF binary
Always call doc.end() If you forget doc.end(), the Promise will hang forever — the 'end' event never fires.

Coordinate System

PDFKit uses a coordinate system where (0, 0) is the top-left corner of the page. X increases to the right, Y increases downward. 

// A4 page is 595 × 842 points (1 point = 1/72 inch)
const pageWidth   = doc.page.width    // 595
const pageHeight  = doc.page.height   // 842
const margin      = 50
const contentWidth = pageWidth - margin * 2  // 495

// Draw text at specific (x, y) coordinates
doc.text("Hello", 50, 100)   // x=50, y=100 from top-left

// Or use the current cursor position (doc.y tracks it)
doc.text("Hello")             // at current cursor position
doc.moveDown()               // move cursor down one line
doc.text("World")             // below Hello

Drawing Text

// Chain: fillColor → fontSize → font → text
doc.fillColor("#333").fontSize(12).font("Helvetica").text("Hello")

// Settings persist until changed
doc.fillColor("red").fontSize(24)
doc.text("Big Red Title")
doc.text("Still red and 24")    // same settings
doc.fillColor("black").fontSize(12)
doc.text("Back to normal")

text() options

// Text with explicit position and width
doc.text("Wrap me", x, y, {
  width:     300,           // wrap text at 300 points wide
  align:     "justify",    // "left" | "center" | "right" | "justify"
  lineGap:   4,             // extra space between lines
  continued: true,          // don't move to next line after this
  characterSpacing: 1.5,   // letter spacing
})

// "continued: true" lets you mix styles on the same line
doc.fillColor("#67CFB1").font("Helvetica-Bold")
   .text("SMART", margin, 22, { continued: true, characterSpacing: 3 })
doc.fillColor("#ffffff")
   .text(" BRIEF", { continued: false, characterSpacing: 3 })

Color hex with alpha PDFKit accepts colors as "#RRGGBBAA" — the last two hex digits are opacity. "#414CC422" = purple at ~13% opacity. This is how the status badge backgrounds are done.

Sending PDF as HTTP Response

// In brief.controller.ts
export const exportBriefDataHandler = async (req, res) => {
  const brief    = await briefService.getBriefById(briefId, auth)
  const usersMap = await fetchUsersFromAuthService([brief.clientId, ...brief.assignedToIds])
  const client   = usersMap.get(brief.clientId) || null
  const assignees = brief.assignedToIds.map(id => usersMap.get(id)).filter(Boolean)

  // Generate the PDF buffer
  const pdfBuffer = await createBriefPdf(brief, client, assignees, auth.role)

  // Send as PDF response
  res.setHeader("Content-Type", "application/pdf")
  res.setHeader("Content-Disposition", `inline; filename="brief-${brief.id}.pdf"`)
  //                                      ↑ "inline" = open in browser
  //                                      "attachment" = force download
  res.status(200).send(pdfBuffer)
}

doc.roundedRect(statusX, titleY, 90, 22, 11).fill(statusColor + "22")
doc.fillColor(statusColor).fontSize(8)
   .text(statusLabel.toUpperCase(), statusX, titleY + 7, { width: 90, align: "center" })

// ── 3. HORIZONTAL DIVIDER ────────────────────────────────────
doc.moveTo(margin, doc.y + 6).lineTo(pageWidth - margin, doc.y + 6)
   .strokeColor("#E2E8F0").lineWidth(0.5).stroke()

// ── 4. SECTION (reusable helper) ────────────────────────────
sectionHeader(doc, "Informations Client", margin)
infoRow(doc, "Nom complet", `${client.firstName} ${client.lastName}`, margin, contentWidth)

// ── 5. BULLET LIST (features) ───────────────────────────────
brief.features.forEach((feature: string) => {
  const y = doc.y                         // capture before drawing
  doc.circle(margin + 4, y + 5, 3).fill("#67CFB1")
  doc.fillColor("#334155").font("Helvetica").fontSize(10)
     .text(feature, margin + 14, y, { width: contentWidth - 14, lineGap: 2 })
})

// ── 6. FOOTER (absolute Y from bottom of page) ──────────────
const footerY = doc.page.height - 44
doc.moveTo(margin, footerY).lineTo(pageWidth - margin, footerY)
   .strokeColor("#E2E8F0").lineWidth(0.5).stroke()
doc.fillColor("#64748B").fontSize(8)
   .text(`Généré le ${new Date().toLocaleDateString("fr-FR")} · Smart Brief`,
        margin, footerY + 8, { width: contentWidth, align: "center" })

Helper Functions

Smart Brief defines two reusable helpers to avoid repeating draw code:

// Draws a labeled section header with a colored underline
function sectionHeader(doc: PDFKit.PDFDocument, title: string, margin: number) {
  doc.moveDown(0.8)
  doc.fillColor("#67CFB1").fontSize(9).font("Helvetica-Bold")
     .text(title.toUpperCase(), margin, doc.y, { characterSpacing: 1.5 })

  const lineY = doc.y + 3
  doc.moveTo(margin, lineY).lineTo(margin + 50, lineY)
     .strokeColor("#67CFB1").lineWidth(2).stroke()

  doc.moveDown(0.6)
}

// Draws a label-value row (like a key-value table)
function infoRow(doc: PDFKit.PDFDocument, label: string, value: string,
                  margin: number, contentWidth: number) {
  const y = doc.y
  doc.fillColor("#64748B").fontSize(9).font("Helvetica")
     .text(label, margin, y, { width: 110 })
  doc.fillColor("#334155").font("Helvetica-Bold").fontSize(9)
     .text(value, margin + 115, y, { width: contentWidth - 115 })
  doc.moveDown(0.4)
}

Usage:

sectionHeader(doc, "Informations Client", margin)
infoRow(doc, "Nom complet", "Omar Ben Ali", margin, contentWidth)
infoRow(doc, "Email",       "omar@example.com", margin, contentWidth)

sectionHeader(doc, "Détails du Projet", margin)
infoRow(doc, "Budget",     "10 000 MAD", margin, contentWidth)
infoRow(doc, "Deadline",   "1 mars 2026", margin, contentWidth)

Role-Based Content in the PDF

The PDF shows different content depending on who is viewing it:

// Client email is hidden from clients (they don't need their own email)
if (client) {
  infoRow(doc, "Nom complet", `${client.firstName} ${client.lastName}`, ...)
  if (role !== "CLIENT") {
    infoRow(doc, "Email", client.email, ...)  // admins + employees see email
  }
}

// Team section: clients don't see who's assigned
if (role !== "CLIENT") {
  sectionHeader(doc, "Équipe Assignée", margin)
  if (assignees.length) {
    assignees.forEach(emp => {
      // Show name + email for each assigned employee
      doc.text(`${emp.firstName} ${emp.lastName} · ${emp.email}`, ...)
    })
  } else {
    doc.text("Non assigné", margin)
  }
}
The role comes from the gateway header auth.role in the controller is read from req.user.role, which was set by requireAuth from the x-user-role header injected by the gateway. So the PDF automatically adapts to whoever makes the export request.
