Microservices

Everything you need to know — concepts, architecture, inter-service communication, and building new services from scratch.

What are Microservices?

A microservice is a small, independently deployable application that does one specific thing. Instead of one big app that handles everything (authentication, data, emails, payments…), you split it into multiple small services that each handle one responsibility.

Each service:

  • Runs as its own process (its own Node.js server)
  • Has its own port and URL
  • Has its own database (no sharing databases between services)
  • Can be deployed, scaled, and updated independently
  • Communicates with other services over HTTP (or message queues)

Monolith vs Microservices

Monolith

One application, one codebase, one deployment, one database.

my-app/
  src/
    auth/        ← user login
    products/    ← product catalog
    orders/      ← order processing
    payments/    ← payment handling
  prisma/
    schema.prisma  ← all tables
  index.ts         ← one server
  • Simple to start
  • Easy to run locally (one command)
  • One deploy = entire app
  • As it grows → hard to scale, slow to test

Microservices

Multiple applications, each with its own codebase, deployment, and database.

services/
  gateway/           ← port 8080
  auth-service/      ← port 3000
  brief-service/     ← port 4000
  (payment-service)  ← port 5000

Each has its own:
  index.ts
  prisma/schema.prisma
  .env
  package.json
  • Each service is small and focused
  • Scale only what needs scaling
  • Deploy auth without touching briefs
  • More complex to set up and run

When to use each

SituationUseWhy
Side project, early-stage app, solo devMonolithFast to build, easy to run locally
Team of 2-4 devs, growing appMonolith or light microModules can be split later
Different parts need different scaleMicroservicesScale the payment service x10, not the whole app
Different parts change at different speedsMicroservicesDeploy auth fixes without risky full redeploy
Learning / internship projectMicroservices (like Smart Brief)Great for understanding real architecture patterns
Microservices are not always better They add real complexity: multiple servers to run locally, CORS between them, network calls between services that can fail, environment variables for each. Start with a monolith unless you have a clear reason to split.

Every request from the frontend goes to the Gateway. The gateway decides which service to forward it to — the client never talks directly to auth-service or brief-service.

The Gateway

The gateway is the single entry point for all traffic. It has two jobs:

  1. Authentication — verify the JWT token on every non-public request
  2. Routing (reverse proxy) — forward the request to the correct service
// services/gateway/src/index.ts — the whole gateway
import { createProxyMiddleware } from "http-proxy-middleware"
import jwt from "jsonwebtoken"

// 1. JWT verification middleware
function authenticate(req, res, next) {
  const isPublic = publicRoutes.some(r => req.path.startsWith(r.path) && req.method === r.method)
  if (isPublic) return next()  // skip auth for public routes

  const token = req.headers.authorization?.split(" ")[1]
  if (!token) return res.status(401).json({ message: "No token" })

  const decoded = jwt.verify(token, JWT_SECRET)
  // Inject user identity as trusted headers
  req.headers["x-user-id"]   = decoded.id
  req.headers["x-user-role"] = decoded.role
  next()
}

app.use(authenticate)  // runs on EVERY request

// 2. Proxy rules — forward by path prefix
app.use("/auth",   createProxyMiddleware({ target: AUTH_SERVICE_URL }))
app.use("/briefs", createProxyMiddleware({ target: BRIEF_SERVICE_URL }))

What createProxyMiddleware does

It forwards the incoming HTTP request to the target service, preserving all headers, body, and method. The response comes back through the proxy to the original client.

// Client sends:  GET /briefs/admin  (to Gateway :8080)
// Gateway sends: GET /briefs/admin  (to Brief Service :4000)
// Brief responds → Gateway returns it to client

app.use("/briefs", createProxyMiddleware({
  target:      process.env.BRIEF_SERVICE_URL,  // http://brief-service:4000
  changeOrigin: true,                         // rewrites the Host header
  pathRewrite: (path) => `/briefs${path}`,   // keeps /briefs prefix
}))
Why pathRewrite? When the client requests /briefs/admin, the gateway strips /briefs from the path before forwarding. With pathRewrite, it re-adds it so the brief service sees /briefs/admin (matching its own router).

Auth Service

Runs on port 3000. Owns everything about users: registration, login, Google OAuth, password reset, profile updates.

// services/auth-service/index.ts
const app = express()
app.use(express.json())
app.use(passport.initialize())

app.use("/auth", authRoutes)
// All routes: POST /auth/register, POST /auth/login,
//             GET /auth/me, PATCH /auth/me, etc.

app.listen(3000)

The auth service has its own authentication middleware for protected routes (like /auth/me). It re-verifies the JWT independently — it doesn't trust the gateway's headers for its own routes.

// Auth service also exposes an INTERNAL endpoint
// for other services to fetch user data
router.get("/api/users/internal", getMultipleUsers)
// GET /api/users/internal?ids=uuid1,uuid2,uuid3
// Returns: [{ id, firstName, lastName, email, role }]
// No authentication required — internal use only

Brief Service

Runs on port 4000. Owns everything about briefs: creation, status updates, assignment, export.

// services/brief-service/src/index.ts
const app = express()
app.use(express.json())
app.use("/briefs", briefRoutes)
app.get("/health", (req, res) => res.json({ status: "OK" }))

app.listen(4000)

The brief service has no JWT verification. It trusts the gateway completely. Its auth middleware just reads the headers that the gateway already injected: 

// services/brief-service/src/middlewares/auth.middleware.ts
export function requireAuth(req, res, next) {
  const userId = req.header("x-user-id")
  const role   = req.header("x-user-role")

  if (!userId || !role) {
    return res.status(401).json({ message: "Unauthorized" })
  }

  req.user = { userId, role }
  next()
}

One Database per Service

Each service has its own separate database. This is a core rule of microservices. Services never share a database — they can only get data from another service by making an HTTP request.

ServiceDatabaseTables
Auth Serviceauth-db (PostgreSQL)User, VerifyToken
Brief Servicebrief-db (PostgreSQL)Brief
Why can't services share a database? If auth-service and brief-service shared one database, they'd be coupled — a schema change in auth would break brief. The whole point of microservices is independent deployment. Separate databases enforce this.

That's why in the Brief model, clientId is just a String — not a foreign key to the User table. The brief service stores the UUID, and when it needs the user's name or email, it makes an HTTP call to the auth service.

How Services Talk to Each Other

There are two main patterns used in Smart Brief:

Pattern 1 — Trusted headers (gateway → downstream service)

The gateway decodes the JWT and injects user info as HTTP headers. Downstream services read those headers and trust them.

When the brief service needs user data (names, emails), it makes an HTTP GET to the auth service's internal endpoint via the gateway.

Trusted Headers Pattern

When the gateway verifies a JWT, it injects the decoded user identity into the request headers before forwarding. Downstream services read these headers instead of decoding JWTs themselves.

Gateway side — inject headers

// services/gateway/src/index.ts
const decoded = jwt.verify(token, JWT_SECRET) as { id: string, role: string }

// These headers are added to the forwarded request
req.headers["x-user-id"]   = decoded.id    // "abc-123-uuid"
req.headers["x-user-role"] = decoded.role  // "ADMIN" | "CLIENT" | "EMPLOYEE"

Service side — read headers

// services/brief-service/src/middlewares/auth.middleware.ts
export function requireAuth(req: Request, res: Response, next: NextFunction) {
  const userId = req.header("x-user-id")    // "abc-123-uuid"
  const role   = req.header("x-user-role")  // "CLIENT"

  if (!userId || !role) {
    res.status(401).json({ message: "Unauthorized" })
    return
  }

  req.user = { userId, role }  // attached to request object
  next()
}

TypeScript: declaring req.user

Because req.user doesn't exist on Express's Request type by default, you declare it using module augmentation:

// Adds req.user to every Express Request in this service
declare global {
  namespace Express {
    interface Request {
      user?: { userId: string; role: string }
    }
  }
}

After this, every controller can access req.user.userId and req.user.role with full TypeScript types.

Internal API Endpoints

An internal endpoint is an API route that is not meant for the client — it exists for service-to-service calls only. In Smart Brief, the auth service exposes: 

// services/auth-service/routes/authRoutes.ts
// Internal API — no authentication required, used by brief-service
router.get("/api/users/internal", getMultipleUsers)
// The handler: GET /api/users/internal?ids=uuid1,uuid2,uuid3
export async function getMultipleUsers(req, res) {
  const { ids } = req.query
  const userIds = ids.split(",").map(id => id.trim()).filter(Boolean)

  const users = await prisma.user.findMany({
    where:  { id: { in: userIds } },
    select: { id: true, firstName: true, lastName: true, email: true, role: true }
  })
  res.json(users)
}
Why no auth on the internal endpoint? This endpoint is called by brief-service, not by the browser. The brief-service doesn't have a user JWT — it's making a server-to-server call. In production, you'd protect this with a shared secret header (e.g. x-internal-secret: mySecret) checked in the handler, or put it in a private network that the public internet can't reach.

Service-to-Service HTTP (authClient.ts)

The brief service has a utility function that wraps all calls to the auth service:

// services/brief-service/src/utils/authClient.ts
import axios from "axios"

export interface UserSnapshot {
  id:        string
  firstName: string
  lastName:  string
  email:     string
  role:      string
}

export async function fetchUsersFromAuthService(
  userIds: string[]
): Promise<Map<string, UserSnapshot>> {

  // Deduplicate and remove empty strings
  const uniqueIds = [...new Set(userIds.filter(Boolean))]
  if (!uniqueIds.length) return new Map()

  const gatewayUrl = process.env.GATEWAY_URL || "http://localhost:8080"

  try {
    const response = await axios.get<UserSnapshot[]>(
      `${gatewayUrl}/api/users/internal`,
      { params: { ids: uniqueIds.join(",") } }
    )
    // Convert array to Map for fast O(1) lookups by id
    return new Map(response.data.map(user => [user.id, user]))
  } catch (error) {
    console.error("Error communicating with Auth Service:", error)
    return new Map()  // graceful degradation — don't crash
  }
}

Using the result in a controller

// Export PDF — needs client + employee names
const userIdsToFetch = [brief.clientId, ...brief.assignedToIds]
const usersMap = await fetchUsersFromAuthService(userIdsToFetch)

// Map gives O(1) lookup — no need to loop to find by id
const clientData = usersMap.get(brief.clientId)      // UserSnapshot | undefined
const assignees  = brief.assignedToIds
  .map(id => usersMap.get(id))
  .filter(Boolean)  // removes undefined for ids not found

// Send brief status email to client
const usersMap = await fetchUsersFromAuthService([updated.clientId])
const client = usersMap.get(updated.clientId)
if (client?.email) {
  await sendBriefStatusEmail(client.email, `${client.firstName} ${client.lastName}`, ...)
}
Why return a Map instead of an array? The Map lets you look up a user by ID in O(1) — usersMap.get(id) — instead of scanning an array every time. Since you often need to look up multiple different IDs from the same response, the Map is much more efficient.

JWT Decoded at the Gateway

The JWT is only decoded once — at the gateway. Every service downstream trusts the headers the gateway injects. This means:

  • Services don't need the JWT_SECRET environment variable
  • Services don't need to install jsonwebtoken
  • JWT verification logic lives in one place
  • If you change JWT format, you update one service
Exception — Auth Service The auth service is special. Routes like GET /auth/me are protected by the auth service's own authenticate middleware, which does re-verify the JWT. This is because some auth routes (like /auth/me) go through the gateway but the auth service still independently verifies — adding a layer of security.

Public Routes

Public routes are endpoints that don't require authentication. They're defined in the gateway so the JWT check is skipped:

// services/gateway/src/index.ts
const publicRoutes = [
  { path: "/auth/register",       method: "POST" },
  { path: "/auth/verify-code",    method: "POST" },
  { path: "/auth/resend-code",    method: "POST" },
  { path: "/auth/login",          method: "POST" },
  { path: "/auth/forgot-password", method: "POST" },
  { path: "/auth/reset-password",  method: "POST" },
  { path: "/auth/google",          method: "GET" },
  { path: "/auth/google/callback", method: "GET" },
]

function authenticate(req, res, next) {
  const isPublic = publicRoutes.some(r =>
    req.path.startsWith(r.path) && req.method === r.method
  )
  if (isPublic) return next()  // skip JWT check
  // ... otherwise verify JWT
}

When adding a new public endpoint, add it to this list.

Trust Boundary

The trust boundary is the line between "the outside world" and "our internal services." The gateway is that boundary. 

External internet
  │
  ▼
┌──────────────────────────────────────────────────────┐
│  GATEWAY (trust boundary)                            │
│  • Verifies JWT from client                          │
│  • Blocks unauthenticated requests to private routes │
│  • Injects x-user-id, x-user-role                   │
└──────────────────────────────────────────────────────┘
  │                        │
  ▼                        ▼
Auth Service           Brief Service
(trusted zone)         (trusted zone)
• Trusts x-user-id     • Trusts x-user-id
• No JWT verification  • No JWT verification
  needed on most routes  needed
Never expose service ports directly to the internet Auth service port 3000 and brief service port 4000 should only be reachable from within the same private network (or from each other). Only the gateway's port 8080 should be public. On Railway, this is done by making internal services non-public.

Adding a Route to the Gateway

Every new service needs a proxy rule in the gateway. Here's the full pattern:

// Step 1: Define any new public routes (if the service has any)
const publicRoutes = [
  // existing routes...
  { path: "/notifications/unread-count", method: "GET" }, // example
]

// Step 2: Add the proxy rule
app.use("/notifications", createProxyMiddleware({
  target:      process.env.NOTIFICATION_SERVICE_URL || "http://localhost:5000",
  changeOrigin: true,
  pathRewrite: (path) => `/notifications${path}`,
  on: {
    proxyReq: (_proxyReq, req) => {
      console.log(`Forwarding: ${req.method} ${req.url}`)
    },
    error: (err) => {
      console.log("Proxy error:", err.message)
    }
  }
}))
Order matters in Express The gateway's proxy rules are matched in order. Put more specific paths before more generic ones. For example, put /api/users/internal before /api/users, or the specific one will never match.

Common Patterns

Graceful degradation on inter-service failure

Network calls between services can fail. Always handle errors gracefully — don't let a failed user-fetch crash the whole request: 

// If auth service is down, still return the brief
try {
  const usersMap = await fetchUsersFromAuthService([updated.clientId])
  const client = usersMap.get(updated.clientId)
  if (client?.email) await sendBriefStatusEmail(...)
} catch (err) {
  console.error("Could not send email", err)
  // Don't throw! The brief was updated successfully.
}

res.status(200).json(updated)  // always return

Health endpoint on every service

// Add this to every service's index.ts
app.get("/health", (req, res) => {
  res.status(200).json({ status: "OK", service: "notification-service" })
})

Hosting providers (Railway, AWS ECS) ping /health to check if the service is alive. If it returns non-200, the service is restarted.

Store only the foreign key — fetch data on demand

// Brief stores clientId (a string UUID)
// NOT a Prisma relation — the User is in a different service
model Brief {
  clientId String   // just the ID — no @relation
}

// When you need the name/email, fetch it at request time:
const usersMap = await fetchUsersFromAuthService([brief.clientId])

Role-based access within a service

// The role comes from x-user-role header
// Controllers enforce role-based access:

export const getAllBriefsHandler = async (req, res) => {
  const auth = req.user
  if (auth.role !== "ADMIN") return res.status(403).json({ error: "Forbidden" })
  // proceed...
}
