Full Stack Web App

Smart Brief

A project brief management platform built for Agence 47. Clients submit project briefs, the admin reviews and assigns them to employees, and employees manage production tasks.

Who Uses It

  • Clients — submit and track briefs, view tasks
  • Admin — accept/refuse briefs, assign employees, create tasks, create clients
  • Employees — manage assigned briefs and update task statuses

Key Features

  • Email + Google OAuth registration
  • Email verification with 6-digit code
  • Role-based UI (3 roles)
  • Task system with progress bar per brief
  • Admin can create walk-in clients & their briefs
  • Email change with verification link
  • PDF brief export
  • Email notifications
Tech stack at a glance: React + TypeScript + Vite + Chakra UI (frontend), Node.js + Express v5 + TypeScript (3 backend services), PostgreSQL + Prisma (database), deployed on Vercel (frontend) and Railway (backend).

Architecture

The backend is split into 3 microservices. The frontend never talks directly to any service — everything goes through the Gateway.

React Frontend (Vercel)
↕
API Gateway :8080 (Railway)
↙      ↘
Auth Service :3000
Users, JWT, OAuth, Email Change
↕ PostgreSQL DB #1
Brief Service :4000
Briefs, Tasks, PDF, Emails
↕ PostgreSQL DB #2

Why a Gateway?

The Gateway is a single entry point for all API calls. It does two critical jobs:

  • JWT Verification — validates the user's token on every request before forwarding it
  • Reverse Proxy — routes requests to the correct service based on the URL path
URL PrefixRouted ToDescription
/auth/*Auth Service :3000Login, register, profile, admin client management
/briefs/*Brief Service :4000All brief and task operations
/api/users/internalAuth Service :3000Internal user lookup (used by Brief Service)

Why Two Databases?

Each service owns its own database. This is a microservices principle called "database per service". The Auth DB stores users; the Brief DB stores briefs and tasks. When the Brief Service needs user details (e.g. a client's name for a PDF), it makes an HTTP call back through the Gateway to the Auth Service.

Redux Toolkit manages global state. Two slices: auth and briefs. The briefs slice also holds task state.

Session Persistence

On login, the JWT is saved to localStorage. On every refresh, App.tsx reads the token, puts it in Redux, then calls getMeThunk to reload the user profile from the server.

Auth Service — What It Does

Standalone Node.js/Express server on port 3000. Handles everything user-related.

Responsibilities

  • User registration with email verification
  • Login with password hashing (bcrypt)
  • JWT token issuance
  • Google OAuth 2.0
  • Password reset via email link
  • Email change with verification link
  • Profile management (get, update, delete)
  • Admin-created clients (walk-in / phone)

Key Libraries

  • bcryptjs — hashes passwords
  • jsonwebtoken — creates/verifies JWTs
  • passport.js — handles Google OAuth
  • Resend — sends emails via HTTP API
  • Prisma — ORM for PostgreSQL

Registration Flow

Two-step process. The user's data is not permanently saved until they verify their email.

1. POST /auth/register
→
2. Save to VerifyToken table
→
3. Send 6-digit code by email
→
4. POST /auth/verify-code
→
5. Create User + return JWT

Password Strength Rules

Passwords must have: minimum 8 characters, at least one uppercase letter, one lowercase letter, and one number. No special character required.

// utils/validation.ts
function isStrongPassword(password: string): boolean {
  if (password.length < 8) return false;
  if (!/[A-Z]/.test(password)) return false;
  if (!/[a-z]/.test(password)) return false;
  if (!/\d/.test(password))    return false;
  return true;
}

Login & JWT

How Login Works

  1. User submits email + password to POST /auth/login
  2. Service looks up the user by email
  3. Uses bcrypt.compare() to check the password against the stored hash
  4. If valid: signs a JWT and returns it along with the user object

What's Inside the JWT

Payload: { id: "user-uuid", role: "ADMIN" } — expires in 7 days. The Gateway verifies this on every request and injects x-user-id and x-user-role headers for the downstream services.

How verification works: The Gateway calls jwt.verify(token, JWT_SECRET). If valid, the decoded id and role are forwarded as custom headers. Services trust these headers without re-verifying the JWT.

Google OAuth 2.0

Users can sign in with their Google account. Uses OAuth 2.0 with Passport.js.

1. User clicks "Sign in with Google"
2. GET /auth/google → redirects to Google consent page
3. Google redirects back to GET /auth/google/callback
4. Passport exchanges the code for a Google profile
5. findOrCreateGoogleUser() — finds by Google ID or email, creates if new
6. JWT signed → redirect to FRONTEND_URL/?token=xxx
7. Frontend reads ?token=, saves to localStorage, removes from URL

Password Reset

1. POST /auth/forgot-password
→
2. Email with reset link
→
3. Click → /reset-password/:token
→
4. POST /auth/reset-password

The link contains a short-lived JWT (30 minutes) with purpose: "reset-password". The token is embedded in the URL path (not a query param) to prevent email clients from stripping it.

Security note: The forgot-password endpoint always returns the same response whether or not the email exists — this prevents attackers from discovering which emails are registered.

Email Change Verification

When a user updates their email in the profile modal, the new email is not applied immediately. Instead:

1. PATCH /auth/me (with new email)
→
2. JWT token generated
→
3. Verification email → new address
→
4. Notification email → old address
→
5. Click link → /verify-email-change/:token
→
6. Email updated in DB
  • The token expires in 1 hour
  • The token is embedded in the URL path (not a query param) for email client safety
  • The old email gets a "your email was changed" notification — if it wasn't them, they can contact support
  • Non-email profile fields (name, phone) are saved immediately without verification
  • The frontend shows an amber banner — "check your inbox" — after requesting the change
  • After confirming, the Redux user store is refreshed so the new email shows everywhere instantly

Admin: Create Clients & Briefs

Not all clients use the website directly — some contact the agency by phone or walk in. The admin can create a client account on their behalf and submit the brief for them.

Creating a Client

Via POST /auth/admin/create-client (admin only). Fields: first name, last name, optional email, optional phone.

  • Email optional — if omitted, a placeholder is auto-generated: firstname.lastname@client.agence47.ma. The isPlaceholderEmail flag is set to true.
  • Password auto-generated — 12-character random password (uppercase + lowercase + digits, no special chars). Returned once in the response — the admin downloads or copies it to give to the client.
  • The client is linked to the creating admin via addedByAdminId.
  • The credentials screen in the UI has copy buttons and a download as .txt button.

Creating a Brief for a Client

Via POST /briefs/admin/for-client/:clientId (admin only). The admin selects a client from their client grid at /admin/new-project, then fills in the same 2-step brief form as a regular client.

The admin's client grid only shows clients they created (addedByAdminId = currentAdmin.id). Clients created by the same admin are listed with their name, email, and phone.

Brief Service — What It Does

Runs on port 4000. Manages briefs, tasks, PDF export, and email notifications for brief events. Trusts the x-user-id and x-user-role headers injected by the Gateway.

Brief Lifecycle

PENDING → ACCEPTED → IN_PROGRESS → COMPLETED
↘ REFUSED
StatusWho Sets ItWhat Happens
PENDINGAuto on creationBrief submitted. Admin notified by email.
ACCEPTEDAdminAdmin accepted. Client notified. Admin can now assign employees.
REFUSEDAdminAdmin rejected. Client notified with optional reason.
IN_PROGRESSAdmin (via assign) / EmployeeSet when admin assigns employees, or employee updates status.
COMPLETEDEmployeeWork marked done. Client notified by email.

Task System

Each brief can have multiple tasks. Tasks track the work breakdown within a brief.

Who Can Do What

  • Admin — create, edit, delete tasks
  • Employees — update task status (if assigned to the brief)
  • Clients — view tasks read-only

Task Fields

  • name — task title
  • description — details
  • priority — LOW | MEDIUM | HIGH
  • startDate / endDate — date range
  • status — PENDING | IN_PROGRESS | COMPLETED

Progress Bar

Both BriefModal and BriefDetails show a progress bar calculated as:

completedTasks / totalTasks × 100%

The bar updates in real time as employees change task statuses.

UI Implementation

The TaskSection component uses Chakra UI v3 for the priority select and date pickers (Select.Root with createListCollection, DatePicker.Root with Portal). This ensures consistent dark theming via CSS [data-part] selectors.

PDF Export

When a user clicks "Export PDF", the frontend calls GET /briefs/:id/export. The Brief Service generates a PDF in memory using PDFKit and streams it to the browser. No files are saved to disk.

Gateway — How It Works

Express server on port 8080 — the single entry point for all client requests.

Processing Order (Every Request)

  1. CORS check — allowed origins: frontend URL, all *.vercel.app, localhost
  2. JWT check — skipped for public routes, verified for everything else
  3. Header injection — x-user-id and x-user-role added to request
  4. Proxy forwarding — request sent to auth-service or brief-service

Public Routes (No Token Required)

POST /auth/register
POST /auth/verify-code
POST /auth/resend-code
POST /auth/login
POST /auth/forgot-password
POST /auth/reset-password
GET  /auth/verify-email-change
GET  /auth/google
GET  /auth/google/callback
GET  /api/users/internal

JWT Verification Flow

Frontend → sends Authorization: Bearer eyJhbG... with every request
↓
Gateway → jwt.verify(token, JWT_SECRET)
↓ invalid → 401    ↓ valid → extracts { id, role }
Gateway → adds headers x-user-id and x-user-role, forwards request
↓
Service → reads headers, sets req.user = { userId, role }
↓
Controller → checks req.user.role to decide access

Prisma ORM

Prisma lets you interact with PostgreSQL using TypeScript. Define your tables in schema.prisma, Prisma generates a fully-typed client.

ConceptWhat It Does
SchemaDefines models (tables) and columns in prisma/schema.prisma
MigrationSQL file describing a schema change. Generated by prisma migrate dev, committed to git.
Deployprisma migrate deploy — applies pending migrations on startup. Runs before the server starts in production.
Generateprisma generate — creates the TypeScript client from the schema. Runs via postinstall.
Railway auto-migration: Each service's package.json start script is prisma migrate deploy && tsx index.ts. Every deployment automatically applies any new migration SQL files before the server starts.

Path-based tokens: Reset and email-change links embed the token in the URL path (/reset-password/TOKEN) rather than a query param (?token=). This prevents certain email clients and mobile proxies from stripping the token.
Free plan limitation: Resend's free plan can only send to verified addresses or addresses on your own domain. To send to any email, add a custom domain (e.g. agence47.ma) to Resend and configure DNS records.

Deployment

Vite builds the React app into static files. Vercel serves them via CDN. Auto-redeploys on every push to main.

A vercel.json catch-all rewrite ensures page refreshes on any route return index.html instead of 404.

3 services (auth, brief, gateway) + 2 PostgreSQL databases — all on Railway. Services communicate via Railway's private internal network.

Each push to main triggers a redeploy. Migrations run automatically on startup.
