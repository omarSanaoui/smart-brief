Prisma ORM

The complete guide — from installation to advanced querying, with real Smart Brief examples.

What is Prisma?

Prisma is a next-generation ORM (Object-Relational Mapper) for Node.js and TypeScript. Instead of writing raw SQL, you interact with your database using type-safe TypeScript methods. Prisma auto-generates a client based on your schema, so your editor knows the shape of every query result.

Without Prisma (raw SQL)

// You write strings, no type safety
const result = await pool.query(
  `SELECT * FROM "User" WHERE id = $1`,
  [userId]
);
// result.rows[0] is `any`

With Prisma

// Full TypeScript types
const user = await prisma.user.findUnique({
  where: { id: userId }
});
// user is User | null — fully typed

Three Parts of Prisma

📄

Prisma Schema

One file (schema.prisma) that describes your database tables, types, enums, and relations. Single source of truth.

⚡

Prisma Client

Auto-generated TypeScript client. Run prisma generate and you get type-safe methods for every model: prisma.user.findUnique(), etc.

🔄

Prisma Migrate

Turns schema changes into SQL migration files. Keeps your database in sync with your schema without losing data.

How they work together You edit schema.prisma → run prisma migrate dev (creates SQL + runs it on DB) → run prisma generate (updates the TypeScript client) → your code now has new types.

Installation

Starting a brand-new project

1

Install dependencies

# Core Prisma tools (CLI + client)
npm install prisma --save-dev
npm install @prisma/client

# For PostgreSQL with connection pooling (what Smart Brief uses)
npm install @prisma/adapter-pg pg
2

Initialize Prisma

npx prisma init --datasource-provider postgresql

This creates two things:

  • prisma/schema.prisma — your schema file
  • .env — with a placeholder DATABASE_URL
3

Set your DATABASE_URL in .env

# Local PostgreSQL
DATABASE_URL="postgresql://user:password@localhost:5432/mydb"

# Railway (what Smart Brief uses)
DATABASE_URL="postgresql://postgres:secret@containers-us-west.railway.app:1234/railway"
4

Write your schema, then generate the client

npx prisma generate

Adding Prisma to an existing project with an existing database

# Pull the existing database schema into schema.prisma
npx prisma db pull

# Then generate the client
npx prisma generate

Prisma inspects your live database and writes the schema.prisma for you — this is called introspection.

schema.prisma

The schema file has three sections:

// 1. Generator — tells Prisma what to generate and where
generator client {
  provider = "prisma-client-js"
  output   = "../src/generated/prisma"  // custom output path
}

// 2. Datasource — your database connection
datasource db {
  provider = "postgresql"
  // DATABASE_URL comes from .env
}

// 3. Models — your tables
model User {
  id        String   @id @default(uuid())
  firstName String
  email     String   @unique
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

Client Singleton Pattern

You should never create a new PrismaClient on every request — it opens a new connection pool each time. Instead, create one instance and export it.

Basic setup (no adapter)

// db/prisma.ts
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

export default prisma;

With @prisma/adapter-pg (Smart Brief setup)

The adapter-pg lets Prisma use an existing pg.Pool connection pool, giving you more control.

// services/brief-service/db/prisma.ts
import "dotenv/config";
import { PrismaClient } from "../src/generated/prisma/index.js";
import { PrismaPg } from "@prisma/adapter-pg";
import pg from "pg";

const pool = new pg.Pool({ connectionString: process.env.DATABASE_URL });
const adapter = new PrismaPg(pool);
const prisma = new PrismaClient({ adapter });

export default prisma;

Auth service (no pool, just adapter string)

// services/auth-service/db/prisma.ts
import { PrismaClient } from "../src/generated/prisma/index.js";
import { PrismaPg } from "@prisma/adapter-pg";

const adapter = new PrismaPg({ connectionString: process.env.DATABASE_URL as string });
const prisma = new PrismaClient({ adapter });

export default prisma;

With logging (useful in development)

const prisma = new PrismaClient({
  log: ["query", "info", "warn", "error"],
  // logs every SQL query Prisma runs — great for debugging
});

Models & Fields

A model maps to a database table. Each line inside is a field (= column).

model User {
  // fieldName  Type  attributes
  id        String    @id @default(uuid())
  firstName String
  phone     String?           // ? = optional (nullable)
  email     String    @unique
  role      Role      @default(CLIENT)   // enum with default
  features  String[]           // array of strings
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

Naming convention

  • Model names: PascalCase singular — User, Brief
  • Field names: camelCase — firstName, createdAt
  • Prisma maps these to snake_case in PostgreSQL automatically (or you can use @@map)

Relations (foreign keys)

Prisma handles relations through special @relation attributes. Here's the full pattern:

model Post {
  id       String @id @default(uuid())
  title    String
  authorId String            // FK column (scalar field)
  author   User   @relation(fields: [authorId], references: [id])
}

model User {
  id    String @id @default(uuid())
  posts Post[]               // back-relation
}
Smart Brief doesn't use Prisma relations The Brief model stores clientId and assignedToIds as plain strings — not as Prisma relations. This is because the User data lives in a different service (auth-service). Cross-service references are strings, not FK constraints.

Define an enum in the schema, then use it as a field type:

// From auth-service/prisma/schema.prisma
enum Role {
  ADMIN
  CLIENT
  EMPLOYEE

  LOCAL
  GOOGLE

  provider Provider @default(LOCAL)
}
// From brief-service/prisma/schema.prisma
enum BriefStatus {
  PENDING
  ACCEPTED
  REFUSED
  IN_PROGRESS
  COMPLETED
}

enum ProjectType {
  SITE_WEB
  SEO
  GOOGLE_ADS
  SOCIAL_MEDIA
  PHOTO_VIDEO
  EMAIL_MARKETING
  COMMUNITY_MANAGER
  BRANDING
  OTHER
}

Prisma turns these into PostgreSQL CREATE TYPE ... AS ENUM and also generates TypeScript enums/union types for you.

// In TypeScript, you can use the enum directly:
import { Role } from "../src/generated/prisma";

const user = await prisma.user.findMany({
  where: { role: Role.ADMIN }
});

// Or as a plain string (Prisma accepts both):
const user = await prisma.user.findMany({
  where: { role: "ADMIN" }
});

Indexes

Indexes speed up queries that filter by a specific field. Define them with @@index:

model Brief {
  id       String      @id @default(uuid())
  clientId String
  status   BriefStatus @default(PENDING)

  @@index([clientId])  // fast lookup by client
  @@index([status])    // fast filtering by status
}

Prisma generates CREATE INDEX statements in the migration file:

-- Generated SQL
CREATE INDEX "Brief_clientId_idx" ON "Brief"("clientId");
CREATE INDEX "Brief_status_idx" ON "Brief"("status");

When writing queries, passing undefined means "don't touch this field." Passing null explicitly sets it to NULL.

// undefined → field not included in query
prisma.user.update({ where: { id }, data: { phone: undefined } });

// null → sets the column to NULL
prisma.user.update({ where: { id }, data: { phone: null } });

prisma migrate dev

The most important command when developing locally:

npx prisma migrate dev --name add_employee_role

What it does:

  1. Diffs your current schema vs the last migration
  2. Creates a new SQL file in prisma/migrations/
  3. Runs that SQL on your local database
  4. Calls prisma generate to update the TypeScript client
The --name flag Always use a descriptive name: --name add_google_auth, --name add_brief_table. This name becomes part of the migration folder name: 20260325_add_google_auth/migration.sql

What a migration file looks like

From the Smart Brief auth-service — when Google OAuth was added:

-- Migration: 20260325091608_init/migration.sql

-- Create the Provider enum type
CREATE TYPE "Provider" AS ENUM ('LOCAL', 'GOOGLE');

-- Add new columns to existing User table
ALTER TABLE "User"
  ADD COLUMN "googleId"  TEXT,
  ADD COLUMN "provider"  "Provider" NOT NULL DEFAULT 'LOCAL',
  ADD COLUMN "updatedAt" TIMESTAMP(3) NOT NULL;

-- Make password nullable (for Google users who have no password)
ALTER TABLE "User"
  ALTER COLUMN "password" DROP NOT NULL;

-- Create unique index on googleId
CREATE UNIQUE INDEX "User_googleId_key" ON "User"("googleId");

Prisma writes this SQL for you — you don't write migration files manually.

prisma migrate deploy

Used in production. Runs any pending migrations that haven't been applied yet. Does NOT create new migrations.

npx prisma migrate deploy

Never run migrate dev on production migrate dev can reset the database to resolve drift. migrate deploy is safe — it only applies forward migrations.

prisma db push

Pushes your schema to the database without creating migration files. Good for fast prototyping or Railway deployments where you don't want to track migration history.

npx prisma db push

migrate dev

  • Creates SQL migration files
  • Tracks full history in git
  • Safe for production via deploy
  • Use for real projects

db push

  • No migration files created
  • Instant schema sync
  • Risk of data loss on breaking changes
  • Use for early prototyping

Migration Files in Detail

All migrations live in prisma/migrations/. Each migration is a folder with a timestamped name:

prisma/
  migrations/
    20260323161115_user/
      migration.sql        ← SQL that creates/alters tables
    20260325091608_init/
      migration.sql        ← adds googleId, provider
    20260401110314_add_employee/
      migration.sql        ← adds EMPLOYEE to Role enum
  schema.prisma

Prisma keeps a _prisma_migrations table in your database that tracks which migrations have been applied.

-- You can see this table in psql:
SELECT * FROM _prisma_migrations ORDER BY finished_at;

create / createMany

create — insert one row

const user = await prisma.user.create({
  data: {
    firstName: "Omar",
    lastName:  "Ben Ali",
    email:     "omar@example.com",
    password:  hashedPassword,
    provider:  "LOCAL",
  },
});
// returns the full User object with all fields

createMany — insert multiple rows

const result = await prisma.user.createMany({
  data: [
    { firstName: "Alice", email: "alice@x.com", ... },
    { firstName: "Bob",   email: "bob@x.com",   ... },
  ],
  skipDuplicates: true,  // skip rows that would violate unique constraint
});
// result = { count: 2 }
// createMany does NOT return the created records, just the count

findUnique / findFirst / findMany

findUnique — exact lookup by unique field

// Must use a field marked @id or @unique
const user = await prisma.user.findUnique({
  where: { email: "omar@example.com" }
});
// Returns User | null
// By primary key
const brief = await prisma.brief.findUnique({
  where: { id: briefId }
});
// Returns Brief | null

findFirst — first matching row

// Finds the first row matching the where clause
const brief = await prisma.brief.findFirst({
  where: { clientId: userId, status: "PENDING" },
  orderBy: { createdAt: "desc" }
});
// Returns Brief | null

findMany — all matching rows

// All briefs for a client, newest first
const briefs = await prisma.brief.findMany({
  where: { clientId: userId },
  orderBy: { createdAt: "desc" },
});
// Returns Brief[]  — empty array if none found
// All briefs assigned to this employee (array contains value)
const briefs = await prisma.brief.findMany({
  where: { assignedToIds: { has: employeeId } },
  orderBy: { createdAt: "desc" },
});

// All users
const allUsers = await prisma.user.findMany();

update / updateMany

update — update one row by unique field

// Must use @id or @unique in where
const updated = await prisma.user.update({
  where: { id: userId },
  data: {
    firstName: "NewName",
    updatedAt: new Date(),  // @updatedAt handles this automatically
  },
});
// Returns the updated User
// Throws PrismaClientKnownRequestError P2025 if record not found

Updating a brief's status (from briefService.ts)

const brief = await prisma.brief.update({
  where: { id: briefId },
  data: {
    status:            status,
    statusReason:      reason?.trim() || null,
    statusUpdatedById: adminId,
    statusUpdatedAt:   new Date(),
  },
});

updateMany — update multiple rows

// Mark all PENDING briefs from a client as REFUSED
const result = await prisma.brief.updateMany({
  where: { clientId: userId, status: "PENDING" },
  data:  { status: "REFUSED" },
});
// result = { count: 3 }
// Does NOT return the updated records

upsert

upsert = UPDATE if exists, INSERT if not. You must provide where, create, and update. 

// From authServices.ts — storing a verification token
// If there's already a pending registration for this email,
// update the code. Otherwise, create a new record.
await prisma.verifyToken.upsert({
  where:  { email },
  update: { code, expiresAt, firstName, lastName, password: hashedPassword, phone },
  create: { email, code, expiresAt, firstName, lastName, password: hashedPassword, phone },
});
// From authServices.ts — Google OAuth user
// If email exists (user registered locally), link their Google account
// If email doesn't exist, create a new user
user = await prisma.user.upsert({
  where:  { email },
  update: { googleId, provider: "GOOGLE" },
  create: {
    firstName: name.givenName,
    lastName:  name.familyName,
    email,
    googleId,
    provider: "GOOGLE",
  },
});
The where clause in upsert must use a unique field Just like findUnique, the where in upsert can only use @id or @unique fields.

delete / deleteMany

delete — one row

// Delete a verify token after the user is created
await prisma.verifyToken.delete({ where: { email } });

// Delete a brief
await prisma.brief.delete({ where: { id: briefId } });

// Returns the deleted record
// Throws P2025 if not found

deleteMany — multiple rows

// Delete all expired tokens
const result = await prisma.verifyToken.deleteMany({
  where: { expiresAt: { lt: new Date() } }  // lt = less than
});
// result = { count: 5 }

Combining conditions

// AND (default — just nest multiple conditions)
prisma.brief.findMany({
  where: {
    clientId: userId,
    status:   "PENDING",
    deadline: { gt: new Date() }
  }
});

// OR
prisma.brief.findMany({
  where: {
    OR: [
      { status: "PENDING" },
      { status: "ACCEPTED" },
    ]
  }
});

// NOT
prisma.brief.findMany({
  where: {
    NOT: { status: "REFUSED" }
  }
});

String filters

// Case-insensitive search across title and description
prisma.brief.findMany({
  where: {
    OR: [
      { title:       { contains: searchTerm, mode: "insensitive" } },
      { description: { contains: searchTerm, mode: "insensitive" } },
    ]
  }
});

// Emails ending with @gmail.com
prisma.user.findMany({
  where: { email: { endsWith: "@gmail.com" } }
});

orderBy & Pagination

orderBy

// Single field
prisma.brief.findMany({ orderBy: { createdAt: "desc" } });

// Multiple fields (sorts by first, then second on tie)
prisma.brief.findMany({
  orderBy: [
    { status:    "asc" },
    { createdAt: "desc" },
  ]
});

Pagination with skip & take

const PAGE_SIZE = 10;
const page = 2;  // 1-indexed

const briefs = await prisma.brief.findMany({
  skip:    (page - 1) * PAGE_SIZE,  // skip 10 rows
  take:    PAGE_SIZE,                // take next 10
  orderBy: { createdAt: "desc" },
});

// Get total count for UI (total pages)
const total = await prisma.brief.count();

select & exclude password

select — choose specific fields

// Only return id, firstName, lastName, email, role
const users = await prisma.user.findMany({
  where:  { role: "ADMIN" },
  select: {
    id:        true,
    firstName: true,
    lastName:  true,
    email:     true,
    role:      true,
    // password: false — omit by not including it
  },
});
// users is now { id, firstName, lastName, email, role }[] — no password field

Exclude password with destructuring (Smart Brief pattern)

Instead of using select, Smart Brief reads the full user then strips the password:

// In authController.ts and authMiddleware.ts
const user = await prisma.user.findUnique({ where: { id: decoded.id } });

// Strip password before sending to client
const { password, ...safeUser } = user;
// safeUser has all User fields EXCEPT password
req.user = safeUser;
select vs destructuring — when to use which Use select when you want to avoid fetching large data from the DB. Use destructuring when you already need the full object but just want to strip a field before sending it to the client.

Transactions

A transaction groups multiple queries — if any one fails, they all roll back.

Sequential transactions (array form)

// All-or-nothing: create user AND delete verify token
const [user] = await prisma.$transaction([
  prisma.user.create({ data: { ... } }),
  prisma.verifyToken.delete({ where: { email } }),
]);
// If create fails, delete never runs
// If delete fails, user is rolled back too

Interactive transactions (callback form)

More powerful — you can include conditional logic inside the transaction:

const result = await prisma.$transaction(async (tx) => {
  // tx is a Prisma client scoped to this transaction
  const brief = await tx.brief.findUnique({ where: { id: briefId } });
  if (!brief) throw new Error("Not found");  // rolls back

  const updated = await tx.brief.update({
    where: { id: briefId },
    data:  { status: "ACCEPTED" },
  });

  return updated;
});
Smart Brief doesn't use explicit transactions verifyAndCreateUser in authServices.ts runs user.create then verifyToken.delete separately. If delete fails after create succeeds, the user exists but the token is stuck. For production systems, this should be wrapped in a $transaction.

Error Handling

Prisma throws typed errors from @prisma/client/runtime:

import { Prisma } from "@prisma/client";  // or from your generated path

try {
  await prisma.user.create({ data: { email: duplicateEmail, ... } });
} catch (error) {
  if (error instanceof Prisma.PrismaClientKnownRequestError) {
    if (error.code === "P2002") {
      // Unique constraint failed
      // error.meta.target = ["email"] (which field)
      throw new Error("Email already in use");
    }
    if (error.code === "P2025") {
      // Record not found (for update/delete)
      throw new Error("Record not found");
    }
  }
  throw error;  // re-throw unknown errors
}

Most common Prisma error codes

CodeWhen it happensCommon cause
P2002create or updateUnique constraint violation (duplicate email, etc.)
P2025update, delete, findUniqueOrThrowRecord does not exist
P2003create, updateForeign key constraint failed
P2000createValue too long for column type
P1001Any operationCan't reach the database server

findUniqueOrThrow / findFirstOrThrow

Instead of checking for null, throw automatically:

// Instead of:
const user = await prisma.user.findUnique({ where: { id } });
if (!user) throw new Error("not found");

// You can write:
const user = await prisma.user.findUniqueOrThrow({ where: { id } });
// throws PrismaClientKnownRequestError P2025 if not found

Prisma Studio

A visual browser-based GUI for your database. Run it locally:

npx prisma studio

Opens at http://localhost:5555. You can:

Real Project Examples

Auth Service — Full user registration flow

// 1. Check if email exists
const existing = await prisma.user.findUnique({ where: { email } });
if (existing) throw new Error("Email already in use");

// 2. Save temp data + send code
await prisma.verifyToken.upsert({
  where:  { email },
  update: { code, expiresAt, firstName, lastName, password: hash, phone },
  create: { email, code, expiresAt, firstName, lastName, password: hash, phone },
});

// 3. Verify code and create real user
const record = await prisma.verifyToken.findUnique({ where: { email } });
if (!record || record.code !== code || record.expiresAt < new Date()) {
  throw new Error("Invalid or expired code");
}

const user = await prisma.user.create({
  data: {
    firstName: record.firstName,
    lastName:  record.lastName,
    email:     record.email,
    password:  record.password,
    phone:     record.phone,
    provider:  "LOCAL",
