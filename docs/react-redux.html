Everything you need to understand how Smart Brief's frontend works — from individual React components to full async data flows with Redux Toolkit and Axios.

What is React?

React is a JavaScript UI library made by Meta. Its job is simple: turn data into HTML on the screen, and update only the parts that changed when the data changes. You do not manually call document.getElementById or touch the DOM — React does that automatically through something called the Virtual DOM.

Virtual DOM — why it matters

Every time your data changes, React builds a lightweight copy of the DOM (the "virtual DOM") in memory and compares it to the previous version. It figures out the smallest possible set of real DOM changes to apply. This is called reconciliation and it is what makes React fast.

Key mental model Think of React as a function: UI = f(state). Given the same state, you always get the same UI. When state changes, React calls your components again and re-renders what changed.

Components

A React component is just a TypeScript function that returns JSX (HTML-like syntax). Components are the building blocks of your app — every button, page, modal, and form is a component. 

// The simplest possible component
function Hello() {
  return <h1>Hello World</h1>;
}

// Used as if it were an HTML tag
<Hello />

Components can be small (a button) or large (an entire page). In Smart Brief, BriefDetails is a full page component and StatusBadgeLarge is a tiny presentational component defined in the same file: 

BriefDetails.tsx
const StatusBadgeLarge = ({ status }: { status: BriefStatus }) => {
  switch (status) {

             </div>;
    case 'ACCEPTED':
      return <div className="... text-sbteal ...">ACCEPTED</div>;
    // ... more cases
  }
}

export default function BriefDetails() {
  // This is the page component — uses StatusBadgeLarge inside
  return (
    <section>
      <StatusBadgeLarge status={brief.status} />
    </section>

}

JSX

JSX looks like HTML but it is actually JavaScript under the hood. The browser does not understand JSX — Vite (your build tool) compiles it into regular React.createElement() calls before the browser ever sees it.

What you write (JSX)
return (
  <button
    className="btn"
    onClick={handleClick}
  >
    Click me
  </button>
);
What it compiles to
return React.createElement(
  "button",
  {
    className: "btn",
    onClick: handleClick
  },
  "Click me"
);

JSX Rules you must know

  • Use className not class — class is a reserved JS keyword
  • JavaScript inside { } — {brief.title}, {isAdmin && <button>...</button>}
  • One root element — every component returns a single root. Use <>...</> (Fragment) to group without adding a DOM node
  • Self-close empty tags — <Clock size={16} /> not <Clock size={16}></Clock>
  • camelCase event handlers — onClick, onChange, onSubmit

Props

Props (short for "properties") are how you pass data into a component — just like arguments to a function. In TypeScript you always type them explicitly. 

// Defining props with a TypeScript interface
interface EditBriefModalProps {
  brief:   Brief;
  isOpen:  boolean;
  onClose: () => void;
}

// The component receives props and uses them
export default function EditBriefModal({ brief, isOpen, onClose }: EditBriefModalProps) {
  if (!isOpen) return null;

  return (
    <div className="modal">
      <h2>{brief.title}</h2>
      <button onClick={onClose}>Close</button>
    </div>
  );
}

// Using it — notice onClose is a callback function passed as a prop
<EditBriefModal
  brief={brief}
  isOpen
  onClose={() => setIsEditOpen(false)}
/>
isOpen without a value Writing <Modal isOpen /> is the same as <Modal isOpen={true} />. A prop present but without a value is implicitly true.

useState

useState is React's hook for local component state. When state changes, React re-renders the component. It returns a tuple: [currentValue, setterFunction]. 

import { useState } from "react";

// TypeScript infers the type from the initial value (boolean here)
const [isEditOpen, setIsEditOpen] = useState(false);

// Open the modal
setIsEditOpen(true);

// Close it
setIsEditOpen(false);

With complex types, you provide the generic explicitly:

// state can be a string or null
const [error, setError] = useState<string | null>(null);

// state is an array of strings
const [features, setFeatures] = useState<string[]>([]);

// state is a full object (type comes from authTypes.ts)
const [formData, setFormData] = useState<RegisterPayload>({
  firstName: "",
  lastName:  "",
  email:     "",
  password:  "",
});

Updating objects in state — do it immutably

React only re-renders if it detects a new reference. You must never mutate state directly. Use the spread operator to create a new object:

// ❌ WRONG — mutates existing object, React won't re-render
formData.email = "new@email.com";
setFormData(formData);

// ✅ CORRECT — creates a new object with the spread operator
setFormData(prev => ({ ...prev, email: "new@email.com" }));

useEffect

useEffect runs side effects — things that happen outside of rendering: fetching data, setting up event listeners, reading from localStorage. It runs after the component renders. 

// From BriefDetails.tsx — fetch the brief when the page loads
useEffect(() => {
  if (id) dispatch(fetchBriefByIdThunk(id));
}, [id, dispatch]);
//  ^^^^^^^^^^^^^^^^^^
//  Dependency array — runs when id or dispatch changes
Dependency ArrayWhen it runs
useEffect(() => { ... })After every single render
useEffect(() => { ... }, [])Once — on mount only (like componentDidMount)
useEffect(() => { ... }, [id])On mount, and whenever id changes

Cleanup

If your effect sets up something that needs tearing down (timers, WebSocket connections), return a cleanup function:

useEffect(() => {
  const timer = setInterval(() => tick(), 1000);
  return () => clearInterval(timer); // cleanup on unmount
}, []);

React Router

React Router handles navigation in a Single Page Application (SPA). Instead of the browser loading a new HTML file on every click, React Router swaps the component being shown based on the URL — no full page reload. 

// useParams — read URL parameters like /briefs/:id
const { id } = useParams<{ id: string }>();

// useNavigate — programmatic navigation
const navigate = useNavigate();
navigate("/dashboard");   // go to a path
navigate(-1);              // go back (like browser back button)

The app's routes are declared in App.tsx using <Route> components, and React Router matches the current URL to the right component automatically.

Why Redux Exists — The Problem

React's useState is perfect for local component state (is this modal open? what is this input's value?). But what happens when many components across the app need the same piece of data?

Imagine the logged-in user object. It is needed on the navbar, the profile page, the brief creation form, the brief details page, and more. Without Redux, you would have to prop-drill — pass the user down through every component in the tree, even ones that don't need it themselves.

→

→

Redux solves this by creating a single global store — a JavaScript object that any component can read from or write to directly, without passing props through the tree.

Redux Core Concepts

Store

The single object holding all your app's state. Created once, lives for the app's lifetime.

State

The data inside the store. In Smart Brief: { auth: {...}, briefs: {...} }

Action

A plain object describing what happened. Has a type string and optional payload.

Reducer

A pure function that takes old state + action, returns new state. Never mutates directly.

Dispatch

The function you call to send an action to the store. Only way to change state.

Selector

A function that reads a specific piece of state from the store for use in a component.

What is Redux Toolkit (RTK)?

Vanilla Redux requires a lot of boilerplate: action type constants, action creator functions, long switch-case reducers, manual TypeScript typing. Redux Toolkit (RTK) is the official, recommended way to write Redux — it eliminates the boilerplate without changing how Redux works.

You need to do thisVanilla ReduxRedux Toolkit
Define action typesManual string constantsAuto-generated by createSlice
Write reducersSwitch/case, manual immutabilityDirect mutation (Immer handles it)
Handle asyncWrite middleware, thunks manuallycreateAsyncThunk
Configure storeapplyMiddleware, compose, DevToolsconfigureStore (all built-in)
Immer — why you can "mutate" in RTK Redux Toolkit uses a library called Immer under the hood. Inside a Redux Toolkit reducer, when you write state.loading = true, you're not actually mutating the state — Immer wraps your operation and produces a new immutable state object. Outside of a Redux Toolkit reducer, you must still follow immutability rules.

configureStore

configureStore is RTK's replacement for Redux's createStore. It sets up the store with good defaults: Redux DevTools extension support and redux-thunk middleware (for async thunks) are included automatically. 

app/store.ts
import { configureStore } from "@reduxjs/toolkit";
import authReducer  from "../features/auth/authSlice/authSlice";
import briefReducer from "../features/briefs/briefSlice/briefSlice";

export const store = configureStore({
  reducer: {
    auth:   authReducer,   // manages state.auth
    briefs: briefReducer,  // manages state.briefs
  }
});

// These types are derived automatically from the store — never write them manually
export type RootState  = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;

RootState and AppDispatch explained

RootState is the TypeScript type of the entire store's state object. ReturnType<typeof store.getState> means "whatever store.getState() returns, that's the type". This is derived automatically — if you add a new slice, the type updates automatically.

AppDispatch is the type of the dispatch function. It is more specific than the default Redux dispatch — it knows about thunks, so TypeScript gives you proper types when you dispatch async actions.

createSlice

A slice is a piece of the Redux store dedicated to one feature. createSlice takes a name, initial state, and reducers, and produces everything you need automatically: the reducer function and action creators. 

authSlice/authSlice.ts
import { createSlice } from "@reduxjs/toolkit";
import type { AuthState } from "./authTypes";

const initialState: AuthState = {
  user:        null,
  token:       localStorage.getItem("token") || sessionStorage.getItem("token"),
  loading:     false,
  error:       null,
  initialized: false,
};
//                                    ↑
// On page load, check if there's a token in storage already.
// This is how "remember me" works — the token survives page reload.

const authSlice = createSlice({
  name: "auth",   // prefix for action types → "auth/logout", "auth/clearError"
  initialState,
  reducers: {
    logout: (state) => {
      state.user  = null;
      state.token = null;
      localStorage.removeItem("token");
      sessionStorage.removeItem("token");
    },
    clearError: (state) => { state.error = null; },
    setToken: (state, action: { payload: string }) => {
      state.token = action.payload;
    }
  },
  // extraReducers handles actions from thunks (see next section)
  extraReducers(builder) { /* ... */ }
});

// Export the action creators so components can dispatch them
export const { logout, clearError, setToken } = authSlice.actions;

// Export the reducer to be registered in the store
export default authSlice.reducer;
What does createSlice generate? For the logout reducer, RTK automatically creates an action creator function logout() that produces { type: "auth/logout" }. You never write action type strings manually.

reducers vs extraReducers

reducers

Used for synchronous actions defined within the same slice. Every key in reducers becomes both a reducer case and an exported action creator.

extraReducers

Used to handle actions that come from outside the slice — mainly from createAsyncThunk. You use the builder pattern to add cases.

Each async thunk automatically creates three action types: .pending, .fulfilled, .rejected. You handle all three in extraReducers:

authSlice/authSlice.ts — extraReducers
extraReducers(builder) {
  builder
    // loginThunk.pending fires when the API call starts
    .addCase(loginThunk.pending, (state) => {
      state.loading = true;
      state.error   = null;
    })
    // loginThunk.fulfilled fires when the API call succeeds
    // action.payload = whatever the thunk returned
    .addCase(loginThunk.fulfilled, (state, action) => {
      state.loading = false;
      state.user    = action.payload.user;
      state.token   = action.payload.token;

      // Persist token based on rememberMe flag sent from the component
      if (action.meta.arg.rememberMe) {
        localStorage.setItem("token", action.payload.token);
      } else {
        sessionStorage.setItem("token", action.payload.token);
      }
    })
    // loginThunk.rejected fires when the API call fails
    // action.payload = the error message (set via rejectWithValue)
    .addCase(loginThunk.rejected, (state, action) => {
      state.loading = false;
      state.error   = action.payload as string;
    });
}
action.meta.arg Inside extraReducers, action.meta.arg gives you the original argument that was passed to the thunk when it was dispatched. In the login example, action.meta.arg.rememberMe is the rememberMe field from the LoginPayload object that the component sent.

Selectors

A selector is a function that reads a piece of state from the store. They live in a dedicated *Selectors.ts file. Why not just access state.auth.user directly in components? Three reasons:

  1. Single source of truth — if you rename a state field, you only update the selector, not every component
  2. Composability — selectors can compute derived values (selectIsAdmin = check user.role)
  3. TypeScript — the selector's return type is inferred, so components get full type safety
authSlice/authSelectors.ts
import type { RootState } from "../../../app/store";

// Simple selectors — just read from state
export const selectUser    = (state: RootState) => state.auth.user;
export const selectToken   = (state: RootState) => state.auth.token;
export const selectAuthLoading = (state: RootState) => state.auth.loading;

// Derived selector — computes a boolean from user data
export const selectIsLoggedIn = (state: RootState) => !!state.auth.token;
export const selectIsAdmin    = (state: RootState) => state.auth.user?.role === "ADMIN";

// Brief selector that takes an extra argument
export const selectBriefById = (state: RootState, briefId: string) =>
  state.briefs.briefs.find(brief => brief.id === briefId);

In a component, you use them with useAppSelector:

// Each call subscribes to exactly the piece of state it needs
const user    = useAppSelector(selectUser);
const loading = useAppSelector(selectAuthLoading);
const isAdmin = useAppSelector(selectIsAdmin);

Typed Hooks

Redux provides two hooks out of the box: useDispatch and useSelector. But they don't know about your store's specific types. Smart Brief wraps them with typed versions in app/hooks.ts: 

app/hooks.ts
import { type TypedUseSelectorHook, useDispatch, useSelector } from "react-redux";
import type { RootState, AppDispatch } from "./store";

// Returns dispatch typed as AppDispatch — knows about thunks
export const useAppDispatch = () => useDispatch<AppDispatch>();

// TypedUseSelectorHook makes useSelector know the shape of RootState
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;

Why useAppDispatch is a factory function: useAppDispatch is () => useDispatch<AppDispatch>() — a function that calls a hook and returns it. You call it inside components like this:

// In a component
const dispatch = useAppDispatch();
// Now "dispatch" is fully typed — TypeScript knows it can handle thunks

dispatch(loginThunk({ email: "...", password: "...", rememberMe: true }));
//        ↑ TypeScript knows this thunk's argument shape

What is a Thunk?

Redux reducers are synchronous and pure — they cannot do async things like make HTTP requests. A thunk is a function that can be dispatched just like an action, but instead of being handled by a reducer directly, it runs custom async logic first, then dispatches real actions.

The word "thunk" is a programming term for a function that wraps another function to delay execution. In Redux, a thunk is a function that returns another function that receives dispatch.

Why middleware? Redux Toolkit includes redux-thunk middleware automatically via configureStore. This middleware intercepts dispatched functions (instead of plain objects) and calls them with dispatch and getState. That's what makes thunks work.

createAsyncThunk

createAsyncThunk is RTK's helper for creating thunks. You give it a type prefix and an async function, and it generates the three lifecycle action creators automatically. 

authSlice/authThunk.ts
export const loginThunk = createAsyncThunk(
//                         ↑ takes two arguments:
  "auth/login",    // 1. Action type prefix (string)

  async (data: LoginPayload, thunkAPI) => {  // 2. Payload creator
    try {
      const response = await api.post("/auth/login", data);
      return response.data; // → becomes action.payload in .fulfilled
    } catch (error: unknown) {
      return thunkAPI.rejectWithValue(getErrorMessage(error, "Login failed"));
      // ↑ → becomes action.payload in .rejected
    }
  }
);

After creating this thunk, you automatically get:

  • loginThunk.pending — action type: "auth/login/pending"
  • loginThunk.fulfilled — action type: "auth/login/fulfilled"
  • loginThunk.rejected — action type: "auth/login/rejected"

Lifecycle Actions — pending / fulfilled / rejected

These three states mirror the lifecycle of a Promise. Here is what happens step by step when a component calls dispatch(loginThunk(...)):

  1. Redux dispatches { type: "auth/login/pending" } immediately
  2. The async function runs (the API call happens)
  3. If the promise resolves → dispatches { type: "auth/login/fulfilled", payload: responseData }
  4. If the promise rejects → dispatches { type: "auth/login/rejected", payload: errorMessage }
// What these action objects look like:

// pending
{ type: "auth/login/pending", meta: { arg: { email: "...", password: "...", rememberMe: true } } }

// fulfilled
{ type: "auth/login/fulfilled", payload: { user: {...}, token: "eyJ..." } }

// rejected
{ type: "auth/login/rejected", payload: "Invalid credentials" }

You handle all three in extraReducers to update the loading/error/data state accordingly.

thunkAPI & rejectWithValue

The second argument to the payload creator is thunkAPI — an object with several powerful utilities.

Method / PropertyWhat it does
thunkAPI.rejectWithValue(value)Reject the thunk with a specific payload instead of the thrown error. Used for user-friendly error messages.
thunkAPI.dispatch(action)Dispatch another action from inside the thunk
thunkAPI.getState()Read the current Redux state from inside the thunk
thunkAPI.signalAn AbortSignal for cancelling the request

The getErrorMessage pattern

Both thunk files in Smart Brief use this helper function to extract clean error messages from Axios errors:

authThunk.ts / briefThunk.ts
const getErrorMessage = (error: unknown, fallback: string) => {
  if (error instanceof AxiosError) {
    // Try to get the message from the server's response body
    return (error.response?.data as { message?: string } | undefined)?.message || fallback;
  }
  return fallback; // Non-axios error (network down, etc.)
};

// Usage inside a thunk's catch block:
return thunkAPI.rejectWithValue(getErrorMessage(error, "Login failed"));
// If server sent { message: "Invalid credentials" }, that's what the user sees
// Otherwise the fallback "Login failed" is used
Why not just throw the error? If you throw inside a thunk instead of using rejectWithValue, the rejected action's payload will be undefined and the error details won't reach your reducer. Always use rejectWithValue to pass clean error data to the .rejected case.

Pattern 4: Dynamic routing based on role

briefThunk.ts
export const updateBriefThunk = createAsyncThunk(
  "briefs/update",
  async ({ id, data, userRole }: UpdateBriefPayload & { userRole?: string }, thunkAPI) => {
    try {
      // Status-only update → use role-specific endpoint
      if (data.status && Object.keys(data).length <= 2) {
        let endpoint = `/briefs/${id}/status`;
        if      (userRole === "ADMIN")    endpoint = `/briefs/admin/${id}/status`;
        else if (userRole === "EMPLOYEE") endpoint = `/briefs/employee/${id}/status`;

        const response = await api.patch(endpoint, { status: data.status, reason: data.statusReason });
        return response.data;
      }
      // Full update → generic PATCH
      const response = await api.patch(`/briefs/${id}`, data);
      return response.data;
    } catch (error: unknown) {
      return thunkAPI.rejectWithValue(getErrorMessage(error, "Failed to update brief"));
    }
  }
);

Pattern 5: DELETE that returns the ID (not the object)

briefThunk.ts
export const deleteBriefThunk = createAsyncThunk(
  "briefs/delete",
  async (id: string, thunkAPI) => {
    try {
      await api.delete(`/briefs/${id}`); // no return value from DELETE

// In briefSlice.ts — the reducer uses the returned id to filter the array
.addCase(deleteBriefThunk.fulfilled, (state, action) => {
  state.loading = false;
  state.briefs  = state.briefs.filter(b => b.id !== action.payload); // action.payload is the id
})

What is Axios?

Axios is an HTTP client for JavaScript/TypeScript. It is used instead of the browser's built-in fetch because it is more convenient: it automatically parses JSON responses, has cleaner error handling, supports request/response interceptors, and handles timeouts.

FeaturefetchAxios
JSON body serializationManual: JSON.stringify(data)Automatic
JSON response parsingManual: await res.json()Automatic: response.data
Error on non-2xx statusNo — must check res.ok manuallyYes — throws automatically
Request interceptorsNo built-in supportYes — great for adding auth headers
Response interceptorsNo built-in supportYes — great for 401 handling
Instance with base URLNoYes — axios.create

axios.create — Creating an Instance

Instead of calling axios.get("https://full-url.com/endpoint") every time, you create an instance with a pre-configured base URL. Smart Brief has two instances — one for the gateway (briefs) and one also used for auth: 

features/briefs/api/briefAxios.ts
import axios from "axios";

const api = axios.create({
  // VITE_API_URL comes from your .env file (the Railway gateway URL in production)
  baseURL: import.meta.env.VITE_API_URL || "http://localhost:8080",
  headers: {
    "Content-Type": "application/json"
  }
});

export default api;

Now in a thunk you write api.get("/briefs") and the full URL becomes https://gateway.up.railway.app/briefs automatically.

Interceptors

Interceptors are middleware for Axios. They intercept every request before it is sent, or every response before it reaches your .then(). Smart Brief uses both:

Request Interceptor — Attach JWT Token

Every API call needs to include the JWT token in the Authorization header. Instead of manually adding it in every thunk, the request interceptor adds it automatically: 

briefAxios.ts
api.interceptors.request.use((config) => {
  // Check both storages — depends on whether "remember me" was ticked
  const token = localStorage.getItem("token") || sessionStorage.getItem("token");

  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
    // Result: every request has the header:
    // Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJpZCI6Ii4...
  }

  return config; // MUST return config or the request won't be sent
});

Response Interceptor — Handle 401 Unauthorized

If a token expires and the server returns 401, the user's session is invalid. The response interceptor catches this globally and redirects to the login page: 

briefAxios.ts
api.interceptors.response.use(
  (response) => response, // Success path — just pass through

  (error) => {            // Error path — called on any non-2xx status
    if (error.response?.status === 401) {
      localStorage.removeItem("token");
      sessionStorage.removeItem("token");
      window.location.href = "/login"; // Hard redirect
    }
    return Promise.reject(error); // Re-throw so thunk catch blocks still fire
  }
);
Promise.reject(error) at the end Even when you handle the 401, you must still call Promise.reject(error) to re-throw the error. If you don't, Axios would treat the error as resolved — your thunk's catch block would never run, and loading would stay true forever.

HTTP Methods with Axios

// GET — fetch data (response in response.data)
const response = await api.get("/briefs");
const response = await api.get(`/briefs/${id}`);

// POST — create (second arg = request body)
const response = await api.post("/auth/login", { email, password });
const response = await api.post("/briefs", briefData);

// PATCH — partial update
const response = await api.patch(`/briefs/${id}`, { status: "ACCEPTED" });

// PUT — full update (replaces entire resource)
const response = await api.put(`/users/${id}`, updatedUser);

// DELETE
await api.delete(`/briefs/${id}`);

// DELETE with a body (deleteMeThunk — sends password for verification)
await api.delete("/auth/me", { data: { password } });

Provider Setup — Connecting Redux to React

For any component to access the Redux store, the entire app must be wrapped with a Provider component from react-redux. This uses React's Context API internally to make the store available everywhere. 

main.tsx
import { StrictMode }  from "react";
import { createRoot }  from "react-dom/client";
import { Provider }    from "react-redux";
import { store }       from "./app/store";
import { BrowserRouter } from "react-router-dom";
import App from "./App";

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <Provider store={store}>        <!-- Redux -->
      <BrowserRouter>             <!-- Router -->
        <App />
      </BrowserRouter>
    </Provider>
  </StrictMode>
);

Provider wraps everything. BrowserRouter wraps everything that needs routing. Your App component is at the center, with its route tree inside.

Login Walkthrough — Line by Line

Let's trace a complete login from form submission to the user appearing on screen.

Step 1 — Component dispatches the thunk

// In a login form component
const dispatch = useAppDispatch();

const handleSubmit = async (e: React.FormEvent) => {
  e.preventDefault();
  await dispatch(loginThunk({ email, password, rememberMe }));
};

Step 2 — pending fires immediately

// extraReducers inside authSlice handles loginThunk.pending
.addCase(loginThunk.pending, (state) => {
  state.loading = true;   // Component reading selectAuthLoading re-renders → show spinner
  state.error   = null;
})

Step 3 — Axios sends the request with the JWT header

// Inside loginThunk payload creator
const response = await api.post("/auth/login", data);
// Before the request is sent, the request interceptor runs:
// config.headers.Authorization = `Bearer ${token}` (if a token exists)
// The request goes to: https://gateway.up.railway.app/auth/login

Step 4a — Success path: fulfilled fires

// Server responds with { user: {...}, token: "eyJ..." }
// Thunk returns response.data
// Redux dispatches loginThunk.fulfilled with that data as payload

.addCase(loginThunk.fulfilled, (state, action) => {
  state.loading = false;
  state.user    = action.payload.user;   // { id, firstName, role, ... }
  state.token   = action.payload.token;  // "eyJ..."
  if (action.meta.arg.rememberMe) {
    localStorage.setItem("token", action.payload.token);
  } else {
    sessionStorage.setItem("token", action.payload.token);
  }
})

Step 4b — Error path: rejected fires

// Server responds with 400 { message: "Invalid credentials" }
// catch block calls: thunkAPI.rejectWithValue("Invalid credentials")
// Redux dispatches loginThunk.rejected with "Invalid credentials" as payload

.addCase(loginThunk.rejected, (state, action) => {
  state.loading = false;
  state.error   = action.payload as string; // "Invalid credentials"
})

Step 5 — Component re-renders with the result

// In the login form component
const error   = useAppSelector(selectAuthError);   // "Invalid credentials" or null
const loading = useAppSelector(selectAuthLoading); // false
const user    = useAppSelector(selectUser);        // User object or null

// In JSX:
{error   && <p className="text-red-400">{error}</p>}
{loading && <Spinner />}
{user    && <Navigate to="/dashboard" />}

The loading/error pattern — every slice follows this

// Initial state always has loading and error
const initialState: SomeState = {
  data:    null,
  loading: false,
  error:   null
};

// Every thunk's three cases follow the same pattern:
.addCase(someThunk.pending,   (state)          => { state.loading = true;  state.error = null; })
.addCase(someThunk.fulfilled, (state, action) => { state.loading = false; state.data = action.payload; })
.addCase(someThunk.rejected,  (state, action) => { state.loading = false; state.error = action.payload as string; })

Awaiting dispatch and checking the result

When you dispatch a thunk with await, it returns an action object. You can inspect it to know if the thunk succeeded:

const result = await dispatch(loginThunk(data));

// Check if it was fulfilled or rejected
if (loginThunk.fulfilled.match(result)) {
  navigate("/dashboard");    // success
} else {
  console.log("Login failed"); // error is already in state.auth.error
}
