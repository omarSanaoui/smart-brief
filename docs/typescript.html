Language Fundamentals

TypeScript

A complete explanation of every TypeScript concept used in Smart Brief — with real code from the project and comparisons to plain JavaScript.

What is TypeScript?

TypeScript is a superset of JavaScript made by Microsoft. "Superset" means every JavaScript file is also valid TypeScript — but TypeScript adds extra features on top, the most important of which is a static type system.

TypeScript does not run in the browser or in Node.js directly. Before it can run, it must be compiled (or transpiled) back to plain JavaScript. In this project, a tool called tsx handles this transparently — you run a .ts file and it just works, no manual compile step needed.

What TypeScript adds

  • Type annotations on variables, parameters, return values
  • Interfaces and type aliases for defining object shapes
  • Generics for reusable typed functions and classes
  • Enums for named constants
  • Compile-time error checking (catches bugs before running)
  • Excellent editor autocomplete and IntelliSense

What TypeScript does NOT change

  • Runtime behavior is identical to JavaScript
  • All type information is erased at compile time
  • No performance difference vs JavaScript
  • All JS libraries work — you just add type definitions
Key idea: Types only exist while you're writing code. By the time the code runs, it's plain JavaScript and has no memory of types. TypeScript is a development tool, not a runtime feature.

TypeScript vs JavaScript

Here's the same function written in both languages:

JavaScript
// No type information
// JS has no idea what types these are
function greetUser(user) {
  return `Hello, ${user.firstName}`;
}

// This crashes at runtime — no warning
greetUser(42);
greetUser({ email: "a@b.com" }); // undefined!
TypeScript
// Types are explicit and checked
interface User {
  firstName: string;
  lastName:  string;
}
function greetUser(user: User): string {
  return `Hello, ${user.firstName}`;
}

// ❌ These are caught BEFORE running:
greetUser(42);             // Error: not a User
greetUser({ email: "" }); // Error: missing fields

The TypeScript version catches both mistakes while you type in VS Code, before the code ever runs. This is the whole point of TypeScript.

Files: .ts and .tsx

There are two TypeScript file extensions used in this project:

ExtensionUsed ForCan it contain JSX?Examples in project
.ts Pure TypeScript — logic, utilities, services, config No authServices.ts, briefSlice.ts, gateway/index.ts
.tsx TypeScript + React (JSX syntax inside) Yes App.tsx, Login.tsx, BriefDetails.tsx

JSX is the HTML-like syntax in React: <div className="card">Hello</div>. The TypeScript compiler needs to know a file might contain JSX so it can handle it correctly — that's what .tsx signals.

Why not always use .tsx? Backend files (services, routes, controllers) are plain .ts because they never contain JSX. Mixing in JSX support where it doesn't belong would be confusing and can cause issues with some tooling.

How TypeScript Actually Runs

In this project, tsx (the npm package, not the file extension) is used as the TypeScript runner for all backend services. It compiles and runs TypeScript on-the-fly — no separate build step needed:

# package.json scripts
"start": "prisma migrate deploy && tsx src/index.ts"
"dev":   "nodemon --exec tsx src/index.ts"

The React frontend uses Vite to bundle and compile the TypeScript frontend into plain JavaScript for the browser.

tsconfig.json

The tsconfig.json file tells the TypeScript compiler how to behave. It lives at the root of each TypeScript project. Key options:

OptionWhat It Does
"target": "ES2022"Which version of JavaScript to compile down to
"module": "ESNext"Module system to use (ES Modules — import/export)
"strict": trueEnables all strict type checks. Most important option.
"jsx": "react-jsx"Tells the compiler how to transform JSX (for React 17+)
"moduleResolution"How to resolve import paths
"outDir"Where to put compiled JS files (when building)

With "strict": true, TypeScript enforces the strictest rules — no implicit any, proper null checks, etc. This is what makes TypeScript actually useful.

Type Annotations

A type annotation is the : TypeName syntax that tells TypeScript what type something is. You can annotate variables, function parameters, and return values.

// Variable annotation
const JWT_SECRET: string = process.env.JWT_SECRET as string;

// Function parameter annotations + return type annotation
function normalizeEmail(email: string): string {
  return email.trim().toLowerCase();
}

// Multiple parameters
async function initiateRegister(
  firstName: string,
  lastName:  string,
  email:     string,
  password:  string,
  phone?:    string  // ← optional parameter
): Promise<{ message: string }> {
  // ...
}

// From brief.controller.ts:
export const updateBriefStatusHandler = async (
  req: Request,
  res: Response
): Promise<void> => {
  // ...
}
Return type annotation: : string after the parameter list says "this function returns a string". : Promise<void> means it's async and returns nothing meaningful (just a promise that resolves).

Type Inference

You don't have to write every type — TypeScript is smart enough to figure out the type automatically from what you assign. This is called type inference.

With explicit annotation
const count: number = 0;
const name: string = "Alice";
const loading: boolean = false;
With inference (same result)
const count = 0;         // inferred: number
const name = "Alice";   // inferred: string
const loading = false; // inferred: boolean

Both are equivalent. In practice, you annotate when TypeScript can't infer (function parameters, complex initial values) and omit the annotation when TypeScript can figure it out.

// In authSelectors.ts — return type inferred automatically:
export const selectUser = (state: RootState) => state.auth.user;
// TypeScript infers the return type is: User | null
// You don't need to write: (state: RootState): User | null =>

export const selectIsLoggedIn = (state: RootState) => !!state.auth.token;
// TypeScript infers return type is: boolean

Union Types — |

A union type means a value can be one of several types. The | symbol reads as "or".

// The most common use: something that can be null
let user: User | null = null;   // either a User object, or null
let token: string | null = null; // either a string, or null

// Union of specific string values (string literal union)
// From authTypes.ts:
role: "ADMIN" | "CLIENT" | "EMPLOYEE"
provider: "LOCAL" | "GOOGLE"

// From briefTypes.ts — the full BriefStatus type:
type BriefStatus = 'PENDING' | 'ACCEPTED' | 'REFUSED' | 'IN_PROGRESS' | 'COMPLETED';

// TypeScript now ensures you can only use these exact values:
const s: BriefStatus = 'ACCEPTED';  // ✅ valid
const s: BriefStatus = 'DONE';      // ❌ Error: not in the union

String literal unions (like 'PENDING' | 'ACCEPTED') are extremely useful for status fields — they act like a lightweight enum that TypeScript enforces at every usage point.

Where you'll see this pattern the most: The role and status fields throughout the entire project. Every time code checks user.role === "ADMIN", TypeScript knows the only valid values and will warn you if you typo it as "admin" or "Admin".

Intersection Types — &

An intersection type means a value must be all of multiple types at once. The & symbol reads as "and". The result is a type that has all properties from both.

// From briefThunk.ts:
async ({ id, data, userRole }: UpdateBriefPayload & { userRole?: string }, thunkAPI) => {
// UpdateBriefPayload & { userRole?: string }
// means: something that has ALL fields of UpdateBriefPayload
//        PLUS an optional userRole string field

// Equivalent to writing out:
type Combined = {
  id:       string;
  data:     Partial<CreateBriefPayload> & { status?: BriefStatus; statusReason?: string };
  userRole?: string;
}

Intersection is useful when you want to extend a type without creating a whole new interface. You're saying "take this existing type and add these extra fields".

Optional Properties — ?

A property marked with ? is optional — it may or may not be present on an object. TypeScript will tell you to check for its existence before using it.

// From authTypes.ts:
interface RegisterPayload {
  firstName: string;   // required — must always be there
  lastName:  string;   // required
  email:     string;   // required
  password:  string;   // required
  phone?:    string;   // optional — might not be provided
  rememberMe?: boolean; // optional
}

// Using an optional property safely:
function formatUser(user: RegisterPayload) {
  console.log(user.firstName);       // ✅ always exists
  console.log(user.phone);           // ✅ but might be undefined
  console.log(user.phone.length);    // ❌ Error! phone might be undefined
  console.log(user.phone?.length);   // ✅ optional chaining — safe
  console.log(user.phone ?? "N/A"); // ✅ nullish coalescing — default value
}

Optional Chaining ?.

This is JavaScript but TypeScript makes it necessary. The ?. operator short-circuits to undefined if the value before it is null or undefined, instead of throwing an error.

// From briefSlice.ts:
if (state.currentBrief?.id === action.payload.id) {
  state.currentBrief = action.payload;
}
// Without ?.  this would crash if currentBrief is null
// With ?.    it safely evaluates to undefined and the if is false

// From authSelectors.ts:
export const selectIsAdmin = (state: RootState) =>
  state.auth.user?.role === "ADMIN";
// user might be null → user?.role returns undefined → not "ADMIN" → false

// From brief.controller.ts:
const adminEmail = process.env.ADMIN_EMAIL || process.env.EMAIL_USER;
// process.env values are string | undefined — handle with ||

Nullish Coalescing ??

// ?? returns the right side only if the left side is null or undefined
const PORT = process.env.PORT ?? 3000;
// If PORT env var is set: use it. If not set (undefined): use 3000.

// vs || which triggers on any falsy value (including "0", "", false)
const PORT2 = process.env.PORT || 3000;
// If PORT is "0" → || would use 3000 (wrong!), ?? would use "0" (correct)

any vs unknown

Both any and unknown mean "I don't know the type". But they behave very differently.

TypeWhat it meansCan you use it directly?Safe?
any TypeScript gives up — no type checking at all Yes — you can call any method, access any property No — bypasses all safety
unknown Could be anything, but you must check before using No — must narrow the type first with typeof / instanceof Yes — forces you to handle it properly 
// From all thunks — catch (error: unknown)
const getErrorMessage = (error: unknown, fallback: string) => {
  if (error instanceof AxiosError) {
    // ✅ Now TypeScript knows it's an AxiosError, gives you .response
    return (error.response?.data as { message?: string } | undefined)?.message
      || fallback;
  }
  return fallback;
};

// Why unknown instead of any?
catch (error: any) {
  console.log(error.message);      // ✅ compiles — but crashes if error has no .message
}
catch (error: unknown) {
  console.log(error.message);      // ❌ Error — must check type first
  if (error instanceof Error) {
    console.log(error.message);    // ✅ Now safe
  }
}

// In the backend controllers — error: any is used for simplicity:
} catch (error: any) {
  res.status(500).json({ message: error.message });
  // This works but is less strict — the backend trusts Error objects
}
Best practice: Use unknown in catch blocks and narrow the type with instanceof. The project uses both — unknown in the frontend thunks (safest) and any in the backend controllers (pragmatic shortcut).

void and never

void

void means a function doesn't return a useful value. It's what you use for async functions that just do work (like sending a response) without returning data the caller needs.

// This function sends a response, it doesn't return anything useful
async function register(req: Request, res: Response): Promise<void> {
  res.status(200).json({ message: "ok" });
  // no return value — that's void
}

never

never means a function never returns at all — either it always throws an error, or it runs forever (infinite loop). It's rare in normal code.

// A function that always throws never returns
function assertUnreachable(x: never): never {
  throw new Error("Should never reach here");
}

// TypeScript uses never internally for exhaustive checks
// and for error states that can't happen

Interfaces

An interface defines the shape of an object — what properties it has and what types they are. It's like a contract: any object of this type must have exactly these fields.

// From authTypes.ts — defines what a User object looks like:
export interface User {
  id:        string;
  firstName: string;
  lastName:  string;
  email:     string;
  phone?:    string;       // optional
  role:      "ADMIN" | "CLIENT" | "EMPLOYEE";
  provider:  "LOCAL" | "GOOGLE";
  createdAt: string;
}

// From brief-service/middlewares/auth.middleware.ts:
export interface AuthContext {
  userId: string;
  role:   UserRole;
}

// From briefService.ts:
export interface CreateBriefInput {
  title:       string;
  projectType: ProjectType;
  description: string;
  features:    string[];
  budgetRange: string;
  deadline:    Date;
  attachments: string[];
}

// Once defined, you use it as a type annotation:
function createBrief(auth: AuthContext, input: CreateBriefInput): Promise<Brief> {
  // TypeScript now knows auth.userId, auth.role, input.title, etc.
}

Interfaces are one of the most important TypeScript features. They make your code self-documenting — you can see exactly what data a function expects without reading the implementation.

Type Aliases

The type keyword creates a type alias — a name for any type expression, including primitives, unions, intersections, and complex objects. Unlike interfaces, they can represent things that aren't objects.

// Simple type aliases for unions — from briefTypes.ts and briefService.ts:
export type BriefStatus  = 'PENDING' | 'ACCEPTED' | 'REFUSED' | 'IN_PROGRESS' | 'COMPLETED';
export type ProjectType  = 'SITE_WEB' | 'SEO' | 'GOOGLE_ADS' | 'SOCIAL_MEDIA' | 'OTHER';
export type UserRole     = "ADMIN" | "CLIENT" | "EMPLOYEE";

// Now you can use these names wherever you need them
const status: BriefStatus = 'ACCEPTED'; // ✅
const status: BriefStatus = 'DONE';     // ❌ Error: not in the union

// Type alias for a complex object (same syntax as interface):
type UpdateBriefPayload = {
  id:   string;
  data: Partial<CreateBriefPayload> & {
    status?:       BriefStatus;
    statusReason?: string;
  };
};

// Function type alias
type RequestHandler = (req: Request, res: Response) => Promise<void>;

interface vs type — When to Use Which

Featureinterfacetype
Define object shapes✅ Primary use✅ Works too
Union types (A | B)❌ Can't✅ Primary use
Intersection (A & B)Via extends✅ Via &
Declaration merging✅ Interfaces can be redeclared and merged❌ Can't
Extend another typeinterface B extends Atype B = A & {...}

In this project: interface is used for object shapes (User, Brief, AuthState, payloads) and type is used for union types (BriefStatus, UserRole, ProjectType). This is the most common convention.

The declaration merging ability of interface is why it's used for module augmentation (see Module Augmentation below) — you can reopen an interface from a library and add fields to it.

Enums

An enum is a named set of constants. TypeScript supports enums natively, and Prisma generates its own enums from the schema.

// TypeScript native enum:
enum Direction {
  Up    = "UP",
  Down  = "DOWN",
  Left  = "LEFT",
  Right = "RIGHT",
}
const dir: Direction = Direction.Up; // = "UP"

// This project doesn't use TypeScript enums directly.
// Instead it uses string literal unions (as type aliases):
type BriefStatus = 'PENDING' | 'ACCEPTED' | 'REFUSED' | 'IN_PROGRESS' | 'COMPLETED';

// And Zod enums for validation in the backend:
export const BriefStatusEnum = z.enum([
  "PENDING", "ACCEPTED", "REFUSED", "IN_PROGRESS", "COMPLETED"
]);

// And Prisma-generated enums for the database:
// (from services/brief-service/src/generated/prisma/enums.ts)
export const BriefStatus = {
  PENDING:     'PENDING',
  ACCEPTED:    'ACCEPTED',
  REFUSED:     'REFUSED',
  IN_PROGRESS: 'IN_PROGRESS',
  COMPLETED:   'COMPLETED',
} as const;

Why string literal unions instead of TypeScript enums?

TypeScript enums have a quirk: they compile to JavaScript objects, which adds runtime code. String literal unions are erased completely at compile time — no runtime overhead. They also play better with JSON and databases since the values are just plain strings.

Generics — <T>

Generics let you write functions, interfaces, and classes that work with any type you specify later. Think of <T> as a type parameter — a placeholder that gets filled in when you use the function.

// A generic function — works with any type
function identity<T>(value: T): T {
  return value;
}
identity<string>("hello");  // T = string
identity<number>(42);     // T = number

// The Map class uses generics:
// From authClient.ts in brief-service:
const map = new Map<string, UserSnapshot>();
// string = key type, UserSnapshot = value type

// From authClient.ts — generic axios response:
const response = await axios.get<UserSnapshot[]>(url);
// Tells TypeScript: response.data is UserSnapshot[]

// Promise is generic:
async function getUser(id: string): Promise<User> {
  // TypeScript knows this async function resolves to a User
}

Generics in Redux Toolkit — createAsyncThunk

The most complex use of generics in this project. createAsyncThunk takes type parameters to know what the thunk returns and what arguments it accepts:

export const loginThunk = createAsyncThunk(

// In the slice — action.payload has the correct type automatically:

Utility Types

TypeScript comes with built-in "utility types" — generic types that transform other types. This project uses several of them.

Partial<T>

Makes every property of T optional. Used when you want to update only some fields of an object.

// From briefTypes.ts:
interface UpdateBriefPayload {
  id:   string;
  data: Partial<CreateBriefPayload> & { status?: BriefStatus; statusReason?: string };
}

// CreateBriefPayload has: title, projectType, description, features, etc.
// Partial<CreateBriefPayload> has: title?, projectType?, description?, features?, etc.
// (all become optional — you only send what you're changing)

Omit<T, Keys>

Creates a type like T but with certain keys removed.

// From authServices.ts:
function excludePassword(user: User): Omit<User, "password"> {
  const { password, ...safeUser } = user;
  return safeUser;
}
// The return type is User without the password field
// TypeScript will error if you try to access .password on the result

// In controllers — same pattern without the explicit Omit:
const { password, ...safeUser } = user;
res.json({ user: safeUser }); // password never reaches the client

Record<K, V>

An object whose keys are all type K and all values are type V. Useful for maps/dictionaries.

// From emailService.ts in brief-service:
const STATUS_LABELS: Record<string, string> = {
  PENDING:     "En attente",
  ACCEPTED:    "Accepté",
  REFUSED:     "Refusé",
  IN_PROGRESS: "En cours de production",
  COMPLETED:   "Terminé",
};
// Record<string, string> = object with string keys and string values
// STATUS_LABELS["ACCEPTED"] returns string

// From authController.ts:
const data: Record<string, unknown> = {};
// An object that can hold any values (used to build update payload dynamically)

ReturnType<T>

Extracts the return type of a function type.

// From store.ts:
export type RootState = ReturnType<typeof store.getState>;
// store.getState is a function
// ReturnType extracts what it returns → the full Redux state object
// This gives you a type with shape: { auth: AuthState, briefs: BriefState }
// Without writing that type out manually — it's derived from the store itself

typeof Operator (TypeScript)

In plain JavaScript, typeof checks the type at runtime. In TypeScript, typeof also works at the type level — it gets the type of a value to use as a type annotation.

// From store.ts:
export type AppDispatch = typeof store.dispatch;
// store.dispatch has a very complex type (dispatch function)
// typeof store.dispatch extracts that type so you can use it elsewhere

export type RootState = ReturnType<typeof store.getState>;
// typeof store.getState → the type of the getState function
// ReturnType<...> → what that function returns = the full state type

// From hooks.ts:
export const useAppDispatch = () => useDispatch<AppDispatch>();
// AppDispatch came from typeof store.dispatch
// So useAppDispatch() knows exactly what actions you can dispatch

This pattern (typeof store.getState and typeof store.dispatch) is the standard Redux Toolkit pattern for TypeScript. It derives your types from the actual store definition, so if you add a new slice, the types update automatically.

Type Assertion — as

Sometimes TypeScript doesn't know the specific type of a value, but you do. as is how you tell TypeScript "trust me, this is of type X".

// From gateway/index.ts:
const decoded = jwt.verify(token, JWT_SECRET) as { id: string, role: string };
// jwt.verify() returns string | JwtPayload — TypeScript doesn't know the shape
// as {...} tells TypeScript: I know the payload has id and role fields

// From gateway/index.ts:
const JWT_SECRET = process.env.JWT_SECRET as string;
// process.env.JWT_SECRET is string | undefined
// as string tells TypeScript: I know it's definitely a string
// (we validated it exists with the if(!process.env.JWT_SECRET) throw above)

// From authSlice.ts — casting unknown Redux payload to string:
state.error = action.payload as string;
// action.payload is typed as unknown by Redux
// We know our thunks always reject with a string message
// as string tells TypeScript to trust us

// From authMiddleware.ts:
const user = req.user as any;
// req.user is Express's generic User type
// as any stops TypeScript from complaining about .id, .role accesses
Warning: as is a promise to TypeScript — it disables type checking for that value. If you lie (as string when it's actually undefined), you get a runtime error. Use it when you genuinely know the type better than TypeScript does, not to silence errors you don't understand.

Non-null Assertion — !

The ! operator after an expression tells TypeScript "I guarantee this is not null or undefined." It's a shorter version of as string for nullable types.

// Without !
const el = document.getElementById("root"); // HTMLElement | null
el.style.display = "none";  // ❌ Error: el might be null

// With !
const el = document.getElementById("root")!; // HTMLElement (null removed)
el.style.display = "none";  // ✅ TypeScript trusts you

// From main.tsx (React entry point):
ReactDOM.createRoot(document.getElementById('root')!).render(...)
// getElementById returns HTMLElement | null
// The ! says: I know this element exists in index.html
Use ! sparingly. Only use it when you're 100% certain the value can't be null at that point. It completely removes null checking — if you're wrong, you get a silent runtime crash.

@ts-ignore

A special comment that tells TypeScript to completely ignore the next line, no matter what errors it has.

// From brief.controller.ts:
const getAuth = (req: Request): briefService.AuthContext => {
  // @ts-ignore
  return req.user;
};
// req.user is typed as Express's generic User type
// but in our app it's actually briefService.AuthContext
// @ts-ignore suppresses the type mismatch error
// The correct solution is module augmentation (see below)
@ts-ignore is a last resort. It hides the error but doesn't fix it. It means you're telling TypeScript "shut up" instead of solving the type mismatch. The alternative — properly augmenting the Express types — was done correctly in the auth middleware (see Module Augmentation).

import type

TypeScript has a special import syntax for importing only types (not runtime values). These imports are completely erased during compilation — they contribute zero bytes to the final JavaScript.

// From authSlice.ts:
import type { AuthState } from "./authTypes";
// AuthState is only a type — it doesn't exist at runtime
// import type makes this explicit and guarantees the import is erased

// From briefSlice.ts:
import type { BriefState } from "./briefTypes";

// From authSelectors.ts:
import type { RootState } from "../../../app/store";

// Compare to regular import (imports a value that exists at runtime):
import { createSlice } from "@reduxjs/toolkit";
// createSlice is a function — it must exist at runtime

import type is a best practice — it makes your intent clear, prevents accidental use of a type as a value, and can improve build performance since bundlers can skip these imports entirely.

TypeScript + Express

Express is a JavaScript library. The TypeScript types for it are in a separate package called @types/express. This package tells TypeScript the shapes of Express's objects like Request, Response, and NextFunction.

// Every route handler in this project uses these Express types:
import { Request, Response, NextFunction } from "express";

// A route handler's type signature:
async function register(
  req: Request,    // the incoming HTTP request (body, headers, params, query)
  res: Response   // the outgoing HTTP response (to send back)
): Promise<void> {
  const { firstName, email } = req.body;  // TypeScript knows req.body exists
  res.status(200).json({ ok: true });      // TypeScript knows .status() and .json() exist
}

Module Augmentation & declare global

This is an advanced TypeScript feature used in this project to add the user property to Express's Request type. By default, req.user doesn't exist on Express's Request type — it's not part of the standard Express API. To tell TypeScript it exists, you extend the type.

// From brief-service/middlewares/auth.middleware.ts:

// This "reopens" the Express module's type definitions
// and adds a new property to the Request interface
declare global {
  namespace Express {
    interface Request {
      user?: AuthContext;  // adds req.user to every Request
    }
  }
}

// Now TypeScript knows req.user is AuthContext | undefined
export function requireAuth(req: Request, res: Response, next: NextFunction) {
  req.user = { userId, role }; // ✅ TypeScript accepts this
  next();
}

Here's what each keyword means:

  • declare global — "I'm adding to the global type namespace"
  • namespace Express — "specifically inside the Express namespace"
  • interface Request — "reopen and extend the Request interface"

This is called declaration merging — TypeScript's ability to merge multiple declarations of the same interface into one. This is why interfaces (not type aliases) are used here — only interfaces can be reopened and merged.

Why this matters: Without this augmentation, every time you access req.user TypeScript would show a red underline. The correct fix is to augment the type (as the brief-service middleware does). The quick fix is as any (as the auth-service uses in its controllers).

Redux Toolkit is designed with TypeScript in mind. The pattern used in this project sets up fully typed state and actions.

The Complete Pattern

// ── 1. DEFINE STATE TYPE ──────────────────────────────
// authTypes.ts
export interface AuthState {
  user:        User | null;
  token:       string | null;
  loading:     boolean;
  error:       string | null;
  initialized: boolean;

// ── 2. TYPED INITIAL STATE ────────────────────────────

}

// ── 3. TYPED THUNK ────────────────────────────────────
// authThunk.ts
export const loginThunk = createAsyncThunk(
  "auth/login",
  async (data: LoginPayload, thunkAPI) => {
    try {
      const response = await api.post("/auth/login", data);
      return response.data; // return type inferred from response
    } catch (error: unknown) {
      return thunkAPI.rejectWithValue(getErrorMessage(error, "Login failed"));
      // rejectWithValue passes a value to the rejected action.payload
    }
  }
)

// ── 4. TYPED STORE ────────────────────────────────────
// store.ts
export type RootState  = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;

// ── 5. TYPED HOOKS ────────────────────────────────────
// hooks.ts
export const useAppDispatch = () => useDispatch<AppDispatch>();
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;

// ── 6. TYPED SELECTORS ────────────────────────────────
// authSelectors.ts
export const selectUser = (state: RootState) => state.auth.user;
// return type automatically inferred as: User | null

// ── 7. USING IN COMPONENTS ────────────────────────────
const user     = useAppSelector(selectUser);    // User | null
const dispatch = useAppDispatch();
await dispatch(loginThunk({ email, password, rememberMe }));

Async functions always return a Promise. TypeScript makes you explicit about what type the Promise resolves to.

// A regular function returning a string:

// From authServices.ts — returns a User (without password field)
async function logUser(email: string, password: string):
  Promise<{ user: Omit<User, "password">, token: string }> {
    const user = await prisma.user.findUnique({ where: { email } });
    if (!user) throw new Error("Invalid credentials");
    const token = jwt.sign({ id: user.id, role: user.role }, JWT_SECRET);
    return { user: excludePassword(user), token };
}

// Error handling in async functions:

  const result = await logUser(email, password);
  // result.user is Omit<User, "password">
  // result.token is string
} catch (error: unknown) {
  // error could be anything — Error, string, object, etc.
  if (error instanceof Error) {
    console.log(error.message); // now TypeScript knows it's an Error
  }
}

TypeScript + Prisma

Prisma generates a fully-typed client from your schema. Every database operation returns proper TypeScript types — you get autocomplete for every field and error checking for every query.

  firstName: string;
