Vite

Next-generation frontend build tool. Dev server + bundler. Used by the entire client/ app.

What Is Vite?

Vite is the tool that powers your npm run dev. It does two things:

  • Dev server — serves your files instantly using native ES modules. No bundling needed during development. Changes are reflected in milliseconds via HMR.
  • Production bundler — uses Rollup under the hood to create an optimized dist/ folder for deployment.

→

→
browser

your src/

Smart Brief uses Vite 8 with React + TypeScript. Entry point: client/index.html → src/main.tsx.

Commands

All run from the client/ directory (or with --prefix client from root).

npm run dev
Start dev server at http://localhost:5173 — hot reload on every save
npm run build
Type-check (tsc) then bundle to dist/ for production
npm run preview
Serve the dist/ folder locally — simulates production before deploy
npm run lint
Run ESLint across all source files
npx vite --port 3000
Dev server on a custom port
npx vite build --mode staging
Build using .env.staging variables
npx vite build --watch
Rebuild on every file change (for library authors)
Dev vs Preview: npm run dev serves raw source via native ESM — no bundle. npm run preview serves the actual dist/ output, which is what Vercel/Netlify will serve. Use preview to catch asset or routing issues before deploying.

index.html is in the root of client/, not in src/. Vite uses it as the application entry point. It contains <script type="module" src="/src/main.tsx"></script>.

vite.config.ts

This is the Smart Brief config exactly as it is:

import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import svgr from 'vite-plugin-svgr'
import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  plugins: [
    react(),        // JSX transform + React Fast Refresh (HMR)
    svgr(),         // import SVGs as React components
    tailwindcss()  // Tailwind v4 processing
  ],
})

The config can accept many more options. Here is the full shape with the most useful ones:

export default defineConfig({
  plugins: [...],

  server: {
    port: 3000,         // change dev port (default 5173)
    open: true,         // auto-open browser on start
    proxy: {            // forward /api calls to backend (avoids CORS in dev)
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true,
      }
    }
  },

  resolve: {
    alias: {
      '@': '/src'        // import Component from '@/components/X'
    }
  },

  build: {
    outDir: 'dist',    // output folder (default)
    sourcemap: true,  // include source maps for debugging prod errors
  },

  base: '/',           // public base path; change for sub-path deploys (/app/)
})

Plugins

@vitejs/plugin-react

Handles JSX transformation and enables React Fast Refresh — the HMR mechanism that preserves component state when you edit a file.

import react from '@vitejs/plugin-react'

react()
// or with options:
react({ babel: { plugins: ['@emotion/babel-plugin'] } })

vite-plugin-svgr

Transforms SVG files into React components. Without this you can only use SVGs as image URLs.

// Without svgr — SVG as URL
import logoUrl from './logo.svg'
const el = <img src={logoUrl} />

// With svgr — SVG as React component (can style with currentColor etc.)
import { ReactComponent as Logo } from './logo.svg'
const el = <Logo className="w-8 h-8 fill-current" />

@tailwindcss/vite

Integrates Tailwind CSS v4 directly into Vite's transform pipeline — no PostCSS config needed. Just call tailwindcss() in plugins.

Environment Variables

Vite exposes env vars through import.meta.env — NOT process.env. Only variables prefixed with VITE_ are exposed to the browser. Everything else stays server-side only.

Never put secrets in VITE_ variables. They are baked into the JS bundle and visible to anyone who opens DevTools. API keys, database URLs, and private tokens belong in server-side env vars only.

The import.meta.env Object

Property Type Value
import.meta.env.VITE_* string Your custom vars from .env files
import.meta.env.MODE string "development" | "production" | custom
import.meta.env.DEV boolean true during dev server, false in build
import.meta.env.PROD boolean true in build, false in dev server
import.meta.env.BASE_URL string The base config option (default "/")
import.meta.env.SSR boolean true only during SSR (not used in Smart Brief)

Smart Brief Usage — VITE_API_URL

// client/.env
VITE_API_URL=http://localhost:8080

// client/src/api/axiosAuth.ts
const AuthApi = axios.create({
  baseURL: import.meta.env.VITE_API_URL || "http://localhost:8080",
  headers: { "Content-Type": "application/json" }
})

On Vercel, VITE_API_URL is set to the Railway gateway URL. At build time, Vite inlines the value into the JS bundle.

Conditional Code Based on Mode

// Show extra debug info in dev only
if (import.meta.env.DEV) {
  console.log('Debug:', data)
}

// Different API URL per mode
const apiUrl = import.meta.env.PROD
  ? import.meta.env.VITE_API_URL
  : 'http://localhost:8080'

Env File Loading Order

Vite loads these files and merges them. Later files win over earlier ones.

File When Loaded Committed?
.env Always Yes — shared defaults
.env.local Always (overrides .env) No — git-ignored, personal overrides
.env.development Only in dev (vite) Yes — dev-specific defaults
.env.development.local Only in dev (overrides .env.development) No — personal dev overrides
.env.production Only in build (vite build) Yes — production defaults
.env.production.local Only in build (overrides .env.production) No — personal prod overrides
.env.staging Only with --mode staging Yes — staging env
Smart Brief setup: Only .env exists with VITE_API_URL=http://localhost:8080. On Vercel, environment variables are set in the dashboard — Vite reads them during the build step.

Example Multi-Environment Setup

# .env (committed — safe defaults)
VITE_APP_NAME=Smart Brief

# .env.development (committed)
VITE_API_URL=http://localhost:8080

# .env.production (committed)
VITE_API_URL=https://gateway.railway.app

# .env.local (git-ignored — override for your machine)
VITE_API_URL=http://localhost:9000  # your custom port

TypeScript Typing for Env Vars

By default, import.meta.env.VITE_* is typed as string | undefined. To get autocomplete and proper types, declare them in a .d.ts file:

// src/vite-env.d.ts  (this file already exists)
/// <reference types="vite/client" />
/// <reference types="vite-plugin-svgr/client" />

// Add this interface to type your custom vars:
interface ImportMetaEnv {
  readonly VITE_API_URL: string
  // add more here as you add VITE_ vars
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}

After this, import.meta.env.VITE_API_URL is typed as string (not string | undefined) and you get autocomplete.

Imports & Static Assets

Importing Images

// Import as URL — Vite hashes the filename on build
import logoUrl from './logo.png'
// logoUrl = "/assets/logo-Bv3a9kY.png" in prod

<img src={logoUrl} alt="Logo" />

Public Folder

Files in client/public/ are served as-is at the root. Not hashed. Use for favicons, robots.txt, etc.

// client/public/favicon.ico → available at /favicon.ico
// Reference in HTML or code as:
<link rel="icon" href="/favicon.ico" />
const url = '/logo.png' // always works, never hashed

Importing SVGs (with svgr plugin)

// As React component (uses ?react suffix or ReactComponent named export)
import { ReactComponent as ArrowIcon } from './arrow.svg'
import ArrowIcon from './arrow.svg?react'   // same, explicit query

// As URL
import arrowUrl from './arrow.svg'

Importing CSS

// Import global CSS in main.tsx
import './index.css'

// CSS Modules — scoped styles
import styles from './Button.module.css'
<button className={styles.primary}>...</button>

Raw Imports

// Import file contents as a string
import shaderCode from './shader.glsl?raw'

// Import as URL string without inlining
import workerUrl from './heavy-worker.js?url'

Path Aliases

Smart Brief doesn't currently use aliases, but they're extremely useful for avoiding ../../../ hell.

// vite.config.ts
import path from 'path'

export default defineConfig({
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src')
    }
  }
})
// tsconfig.json — also add this for TS to understand the alias
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
// Before alias
import { Button } from '../../../components/ui/Button'

// After alias
import { Button } from '@/components/ui/Button'

Build & Preview

# Run type checking then build
npm run build

# Output:
# dist/
#   index.html
#   assets/
#     index-Cv3a9kY.js   ← hashed for cache busting
#     index-BH3b8nX.css
#     logo-Av9bK3m.png

What build does:

  • Runs tsc -b for TypeScript type checking first (will error on type issues)
  • Tree-shakes unused code
  • Minifies JS with esbuild and CSS with lightningcss
  • Hashes asset filenames for long-term caching
  • Inlines small assets (<4kb) as base64
# Preview the built output locally
npm run preview
# Serves dist/ at http://localhost:4173
# Use this to check routing, 404s, env vars before deploying
On Vercel: The build command is npm run build, output directory is dist. Vercel sets NODE_ENV=production automatically, so Vite uses .env.production vars during the build.

Hot Module Replacement (HMR)

During dev, when you save a file:

  • Only the changed module and its direct importers are re-evaluated
  • React component state is preserved (Fast Refresh)
  • CSS changes are applied instantly without a page reload
  • Non-React JS modules do a full page reload
State preservation caveat: If you edit a file that exports both a React component and a non-component (like a hook + a constant), Fast Refresh may fall back to a full reload. Keep components in their own files for best HMR behavior.
